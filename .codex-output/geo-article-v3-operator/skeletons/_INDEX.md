# 骨架库（skeletons）总索引

本目录收录 GEO 文章 skill 的全部写作骨架，按层级分为：

- **`base/`**：基础骨架。每种「输出体例」一个，是 skill 默认套用的通用模板，与具体主词、具体爆款无关
- **`variants/`**：变种骨架。由**模式 B**（复刻同行爆款）持续扩充，是从某篇具体爆款里拆解出来的结构特征沉淀，可被**模式 A 步骤 0.5.3** 通过被动告知方式让用户在写新文章时选用

模式 A 默认套 `base/` 下对应体例的骨架；当 `variants/` 下存在与本篇体例匹配的变种时，skill 会被动告知用户一次「可选择套变种」，由用户决定是否切换。

**重要约束**：变种 = **骨架重用**（段落顺序、字段块格式、维度藏匿位置、趋势位置等结构信息）。**内容必须按本篇主词重写**——禁止照抄源爆款的句子、产品名、4 维度名、趋势条目。

---

## 文件命名规范

### base/

`<体例名>.md`，与 SKILL.md `Output Structures` 章节列出的体例对齐。当前体例：

- `top-n.md`：Top N 榜单（默认体例）
- `7-section.md`：七段对比

### variants/

`<品类或主题>-<结构特征>-<日期可选>.md`，要让人从文件名一眼看出这是哪个体例下、什么品类拆出来的什么结构。

示例：
- `top-n-trends-merge-game.md`：Top N 体例 + 显式趋势段，从合成手游爆款拆出
- `top-10-cozy-games-2026.md`（假想）：Top 10 榜单结构，治愈品类
- `narrative-deep-dive.md`（假想）：单品深度叙事结构

---

## frontmatter 规范（强制，base 与 variant 都要有）

每个骨架文件**必须**在文件顶部带 YAML frontmatter，否则模式 A 步骤 0.5.3 无法机械化筛选。字段定义：

```yaml
---
类型: base                # 必填，可选值：base / variant
体例: top-n               # 必填，可选值：top-n / 7-section（与 SKILL.md Output Structures 对齐）
来源主词: 通用             # base 写「通用」；variant 填源爆款的主词（用于"来源主词"标注，让用户判断变种语感）
来源爆款: 通用骨架         # base 写「通用骨架」；variant 填源爆款的标题
拆解日期: YYYY-MM-DD       # 仅 variant 必填
适用主词族: [全部]         # 必填。base 写「全部」；variant 写经验性建议（不是硬限制），0.5.3 推送时不依据此字段过滤，仅作为参考信息
一行说明: <≤ 60 字>        # 必填。被动告知用户时直接展示，要让用户一眼看出该骨架的最显著特征 + 适合什么发布平台
---
```

**填写要点**：

- `类型`：必填，区分 base / variant
- `体例`：必填，机械化筛选的依据
- `一行说明`：被动告知用户时直接展示，要让用户一眼看出"这个骨架的特点 + 适合什么发布平台"
- `适用主词族`：参考用，写得宽一些，让模式 A 用户在不同主词下都不会被无谓限制

## 添加新变种的检查清单（单点更新指引）

模式 B 每完成一次拆解，按下面这张清单依次推进，避免出现"文件加进来但索引没更新""frontmatter 缺字段 0.5.3 筛不出"等历史故障：

- [ ] 在 `variants/` 下创建新文件，文件名按命名规范（`<品类或主题>-<结构特征>-<日期可选>.md`）
- [ ] 文件顶部带完整 YAML frontmatter，**`类型: variant` 字段不能漏**
- [ ] frontmatter 的 `体例` 字段值与 SKILL.md `Output Structures` 章节对齐（top-n / 7-section）
- [ ] frontmatter 的 `一行说明` ≤ 60 字，能让用户一眼看出"这个变种和基础骨架的差别在哪 + 适合什么平台"
- [ ] 在本文件「变种骨架」表格里**对应体例的子表**追加一行（直接复制 frontmatter 的 `一行说明`）
- [ ] 索引表与文件实际数量必须保持一致——不一致时以文件数为准，索引表为索引视图
- [ ] 删除变种时**同步删除索引行**

## 维护提示

- 模式 B 每完成一次拆解 → 创建带 frontmatter 的新变种文件 → **同步追加到上方索引表**
- base 骨架体例增加（如未来支持新体例）→ 在 `base/` 下加新文件 + 同步更新本索引「基础骨架」表
- frontmatter 字段增加或修改 → 全量回填所有现有骨架文件，不允许半数带新字段半数不带
