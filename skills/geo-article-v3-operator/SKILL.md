# GEO Article v2

## Goal

围绕《餐厅养成记》产出 GEO 文章。两个使用入口：持续生成（模式 A）、复刻同行爆款（模式 B）。两种输出体例：Top N 榜单（默认）、七段对比。

## Output Structures（输出体例）

skill 支持两种文章体例，模式 A 与模式 B 都可使用：

### 体例 1：Top N 榜单（**默认**）

骨架：[`skeletons/base/top-n.md`](skeletons/base/top-n.md)

适用：发布平台对软文识别较严的渠道（公众号、头条、知乎等）。结构上像普通玩家整理的推荐榜单，规避显式「评估维度段 / 趋势洞察段 / 主推深写 vs 陪衬收写」这类典型软文特征。形态：1 主推（Top 1）+ N-1 其他产品平铺，每款独立成段。

### 体例 2：七段对比

骨架：[`skeletons/base/7-section.md`](skeletons/base/7-section.md)

适用：内容质量优先、不在意发布平台严审的场景；或用户明确要求深度对比。形态：标题 → 引言 → 评估维度说明 → 主体分项展开（主推深写 + 陪衬收写） → 趋势洞察 → 人群分流推荐 → 总结收口。

### 体例切换（关键词识别）

skill 在用户指令里识别这些短语：

| 用户说 | 用哪个体例 |
|---|---|
| 「Top N 榜单」「Top N」「榜单结构」「推荐榜」「精选推荐」 | Top N 榜单 |
| 「七段结构」「七段对比」「深度对比」「主推深写」 | 七段对比 |
| 都不指定 | **默认 Top N 榜单** |

## Operating Modes

### 模式 A：持续生成（规划可选）

适用：用户日常持续产出 GEO 文章。1 篇 / 5 篇 / 10 篇都可以，无固定数量。

**默认主词**：模拟经营。每次用户可临时覆盖（如「这次主词改成『治愈手游』」）。

#### 步骤 0（首次使用）：实例化日志文件

如果 `assets/article-plan.md` 不存在，复制 `assets/article-plan-template.md` → `assets/article-plan.md`。这是后续滚动日志和差异化检查的基础。

#### 步骤 0.4：并行预读（性能优化，必跑）

**目的**：把"边写边读"改成"开篇一次性把所需资料全拉到上下文里"，避免串行 Read 拖慢端到端时长。实测能省 20-30 秒、减少 4-6 次工具往返。

**执行方式**：在同一条消息里**并行**发起以下 Read 调用（一次性全发，不要等结果回来再发下一个）：

| 文件 | 取段策略 |
|---|---|
| `assets/article-plan.md` | 全文（差异化检查 + 已写日志） |
| `references/companion-pool.md` | 全文（库召回） |
| `references/product-knowledge.md` | 全文（主推事实 + §10 维度 + §12 综合画像 + §13 同厂清单） |
| `references/writing-guide.md` | offset 取 §10 标题方法论 + §11 差异化 + §13 宽窄判定 三个区段（用 Grep 先定位行号也可，但 Read offset/limit 更省一次往返） |
| `skeletons/_INDEX.md` | 全文（变种推送依据） |
| `skeletons/base/top-n.md` 或 `skeletons/base/7-section.md` | 按 0.5.1 体例决定的那一份。如果体例还没定，**两份都读**（成本可忽略） |

**关键约束**：
- 必须在**一个**消息里并行发起，不允许串行
- 读完后**所有后续步骤（0.5 / 2 / 2.5 / 3）禁止再次 Read 这些文件**，除非内容确实超出预读范围
- 写作时直接从已读上下文取素材，不再回头读骨架/维度表/产品知识
- 例外：用户在 2.5.7 把某陪衬替换成库外候选时，可允许补一次 WebSearch

**为什么并行**：每次 Read 都有~3-5 秒的往返开销，6 次串行 = 18-30 秒；6 次并行 = 5-8 秒（取最慢一次）。这是端到端速度最容易拿到的红利，没有副作用。

#### 步骤 0.5：确定本篇输出体例 + 主词宽窄判定 + 结构变种推送 + 提醒免责声明

**0.5.1 体例确定**

按上方「体例切换（关键词识别）」规则确定本篇用 Top N 榜单还是七段对比。

**0.5.2 主词宽窄判定（必跑）**

按 [`references/writing-guide.md`](references/writing-guide.md) §13.1 二元判定规则，判断本篇走哪个分支：

- **常规分支**：主推《餐厅养成记》能锁定主推复合赛道代表位 → 按常规流程
- **跨赛道分支**：主词太宽（如「游戏」「手游」），主推无法代表所有产品 → 主推承担「合成+经营+剧情复合赛道头部」单点位，陪衬各占一个完全异形态赛道。陪衬筛选规则、产品定位段写法、第一条分流写法均按 §13.3 调整

10 个目标主词的预标记见 §13.4。清单外的主词按 §13.1 现场判分；模棱两可的主动询问用户（§13.6）。

**0.5.3 结构变种推送（被动告知）**

体例确定后，读 [`skeletons/_INDEX.md`](skeletons/_INDEX.md) 顶部的「变种骨架」节。按 0.5.1 确定的体例筛出 `frontmatter.体例 == 本篇体例` 的所有变种文件。

- **若该体例下没有可用变种** → 本步骤跳过，直接套基础骨架，不告知不打扰
- **若有 ≥1 个匹配变种** → **被动告知**用户一次：

  > 「[本篇体例] 体例下当前可用的结构变种：
  > - `<变种文件名>`：<frontmatter.一行说明>（来源主词：<frontmatter.来源主词>）
  > - ...
  >
  > 本篇默认套基础骨架，需要套变种请告诉我用哪个。」

- 用户**没回应** / **说不用** → 本篇套基础骨架
- 用户**点名某变种**（如「用 top-n-trends-merge-game」「套合成手游高留存那个」）→ 本篇套该变种文件

**关键约束（变种 = 骨架重用，内容必须按主词重写）**：

- 变种文件提供的是**段落顺序、字段块格式、维度藏匿位置、趋势位置**等结构信息
- **禁止照抄源爆款的句子、产品名、4 维度名、趋势条目**——这些必须按本篇主词与副词组重新生成
- 模式 A 借用变种时，「不复用源爆款句子」与模式 B 的同名硬约束**等价生效**

提醒只一次，不重复，不阻塞流程。

**0.5.4 免责声明提醒**

**用户偏好检查**：先查 `assets/article-plan.md` 顶部「当前默认主词、陪衬数量、体例」节里是否有「**默认免责声明**：加 / 不加」一行。
- 有「加」 → 本篇直接加，跳过提醒
- 有「不加」 → 本篇直接不加，跳过提醒
- 无该行 / 写「按需提醒」→ 按原规则提醒一次

**提醒模板**（仅在用户未配置默认时使用）：「本篇默认**不加**免责声明，需要加请告诉我」。

- 用户**没回应** / **明确说不加** → 本篇不加免责声明
- 用户**说要加** / **「加一下」** → 本篇末尾加免责声明（参考 [`skeletons/base/top-n.md`](skeletons/base/top-n.md) §段 7 模板）

提醒只一次，不重复，不阻塞流程 —— 等用户继续发指令（如「写一篇」「先规划」）就直接按用户决定执行。

**0.5.5 合并询问优化（推荐）**

若 0.5.3 有变种需要被动告知 + 0.5.4 需要提醒免责声明 + 用户启动指令含糊（如「写一篇」未指定主题角度），**强烈建议把这三件事合并到一次 AskUserQuestion 里**——一次性把"角度 / 变种 / 免责声明"丢给用户拍板，省 1-2 次往返。

合并条件：
- 用户启动指令为「写一篇」「生成一篇」「随便写一篇」等未指定具体角度的话
- 本体例下有 ≥1 个可用变种
- `assets/article-plan.md` 顶部未配置默认免责声明

满足上述条件 → 用 AskUserQuestion 同时问 2-3 个问题（主题角度 / 是否套变种 / 是否加免责声明），用户一次性回答完进 Step 2。

不满足条件（如用户已指定角度 / 无变种 / 已配置默认）→ 该跳过的跳过，不要为了"合并"而硬塞问题。

#### 步骤 1（可选）：先规划

仅当用户明确说「先规划一下」「先列个表」时执行：
- 读 `references/product-knowledge.md` + `references/writing-guide.md`
- 在 `assets/article-plan.md` 的「待写区」追加 N 行（含「体例」列）
- **停下**，等用户说「写第 X 篇」或「直接写一篇」

如果用户上来直接说「写一篇」，**跳过此步**，直接进步骤 2。

#### 步骤 2：写每篇前的差异化检查（必查）

读 `assets/article-plan.md` 「已写日志」节最近 1-3 行：
- 主题角度**不可相同**
- 高权重副词组与上一篇重合度 **≤ 50%**
- 陪衬组合与上一篇重合度 **≤ 50%**（陪衬重合检查在步骤 2.5 之后才能完整执行，此处先检查主题角度和副词）

差异化检查**跨体例**进行 —— 即使本篇用 Top N、上一篇用七段，主题角度 / 副词 / 陪衬组合的重合度依然按规则比较。

不达标 → 提示用户具体重合在哪 + 给出调整建议 → 等用户确认调整方案 → 再继续。
达标 → 直接进步骤 2.5。

#### 步骤 2.5：陪衬筛选（库优先 → WebSearch 兜底）

**核心思路**：陪衬经过 2.5.7 用户明确确认后沉淀进 [`references/companion-pool.md`](references/companion-pool.md)。下次写作时**先查库**，库内能凑齐就跳过 WebSearch，凑不齐才补。

**为什么这么改**：每篇都跑 2-3 条 WebSearch query 太耗时间，但又不能退回经验池（训练数据里混营销文虚构作品 / 停服老作 / 译名混乱）。库里的每一条都是"用户审过的真陪衬"——复用它们既快又安全，唯一要防的是停服（180 天重验机制兜底）。

完整流程详见 [`references/companion-games.md`](references/companion-games.md) §6。

##### 2.5.0：工具可用性检查（仅冷启动 / 库不足时必跑）

**WebSearch 仍是首次接触陪衬时的硬依赖——经验池/LLM 记忆不能替代**。原因：

- LLM 训练数据里包含大量营销文虚构的"AI 时代神作"，无法分辨真假
- 同款游戏在中/英/日存在多个译名，靠记忆容易混淆产品名
- 训练数据里仍存在已停服的游戏，LLM 无法判断当前运营状态
- 单会话内 LLM 推荐会重复偏好同几款熟悉作品，违反差异化策略

但**不再每篇都强制跑** —— 库里已审过的陪衬可直接复用（首次入库时已经 WebSearch 验过一次），只要不超过 180 天重验阈值。

操作流程：

1. **先跑 §2.5.0.5 库召回** —— 若库内能凑齐 N 款匹配候选，跳过 WebSearch 直接进 §2.5.7（**热启动**）
2. **库不足 N 款 → 必须跑 WebSearch**（**冷启动 / 半热启动**）：
   - 检查 WebSearch 是否在当前 functions 列表 —— 在则进 §2.5.1
   - 不在则用 ToolSearch 加载：`ToolSearch("select:WebSearch")`
   - 加载也失败 → **不要凑陪衬**，告知用户：
     > 「本篇陪衬筛选所需的 WebSearch 工具当前不可用，且库内匹配候选不足 N 款。请选择：
     > ① 等环境恢复后重试
     > ② 你口头给一份陪衬名单，我用 6 条硬规则校验后入文
     > ③ 取消本篇
     > ④ 放宽匹配条件、只用库内现有候选（库内候选 < N 款时不推荐，差异化和细分覆盖会受损）」

**为什么禁止经验池补位**：凑出来的陪衬质量极低，会让 §2.5.7 用户确认成为"每次都被打回"的常态。2.5.7 是兜底，不是治本，不应被滥用。库优先机制不绕过这条 —— 库里的每条都是 WebSearch 命中 + 用户确认后才进的，不是凭记忆。

##### 2.5.0.5：库优先召回（必跑，热启动入口）

**目的**：把 2.5 从"每篇必跑 WebSearch"改成"先查库，凑不齐再补 WebSearch"，省掉 70% 重复搜索时间。

操作流程：

1. **读 [`references/companion-pool.md`](references/companion-pool.md)**，按本篇主词 + 副词组的赛道方向筛选候选：
   - 匹配条件：`适用主词` 列命中本篇主词 ∪ `单点方向` 与本篇副词组对齐
   - **信源强度 = 强** 优先入选；中 次之
   - **首次确认日期 > 180 天** 的条目 → 标记"待重验"，**不直接采用**，按 §2.5.0.5.重验流程处理
   - **多赛道副本去重**：库里同款游戏可能在多个赛道分组各有一行（卖点角度不同）。召回时若同一游戏名命中多行，**只保留与本篇副词组最贴合的那一行**，其他副本丢弃；这条游戏在本篇里只算 1 款候选，不算多款

2. **应用 6 条硬规则过滤**（与 WebSearch 候选用同一套规则）：
   - 同厂排除（查 [`references/product-knowledge.md`](references/product-knowledge.md) §13.1.B）
   - 危险竞对排除
   - 单点方向不扎堆（库内候选若同方向 ≥ 2 款，保留单点最鲜明的 1 款）
   - 跨赛道分支按 [`references/writing-guide.md`](references/writing-guide.md) §13.3 调整

3. **剔除最近 3 篇已用陪衬**（查 `assets/article-plan.md`「已写日志」最近 3 行的「陪衬组合」列）—— **按游戏名去重**，即同款游戏即使本篇召回的是"另一赛道副本"，仍视为最近用过，必须剔除。不允许靠切换赛道副本绕过排重

4. **判定召回结果**：

   | 库内可用候选数 | 动作 |
   |---|---|
   | ≥ N（默认 4） | **热启动** —— 直接挑细分方向最分散的 N 款 → 进 §2.5.7 用户确认（跳过 §2.5.1-2.5.6 WebSearch） |
   | 1 ≤ X < N | **半热启动** —— 库内 X 款保留为基底，跑 §2.5.1-2.5.6 WebSearch 补 (N-X) 款新血 |
   | 0 | **冷启动** —— 完全跑 §2.5.1-2.5.6 WebSearch，与改动前一致 |

5. **重验流程**（库条目首次确认日期 > 180 天时）：
   - 单 query WebSearch：`"<游戏名> <当前年份> 是否在运营"` 或类似
   - 命中且仍在运营 → 该候选进 2.5.7 让用户重新确认 → 通过后**刷新该条目的「首次确认日期」为今天**
   - 命中显示停服 / 改名 / 长期未更新 → 候选剔除，2.5.7 时告知用户该条目状态变化

##### 2.5.1-2.5.6：WebSearch 筛选执行流程（冷启动 / 半热启动时跑）

**触发条件**：§2.5.0.5 召回后库内候选 < N 款，需要补差额或全新筛选。

简化执行：

1. 基于本篇主词 + 副词组 + 评估维度，生成 **2-3 条搜索 query**（半热启动时可减为 1-2 条，因为只补差额）
2. WebSearch **各取 top 5**，合并去重得候选池
3. 按 [`references/companion-games.md`](references/companion-games.md) §2 **6 条硬规则**过滤：
   - ① 同属大品类
   - ② 用户都认识 / 细分赛道有代表性
   - ③ 危险竞对排除（与主推 ≥3 项高度重合 → 剔除）
   - ④ 长板鲜明、短板也明显（剔除全面均衡型）
   - ⑤ 整组细分方向不能重复（**含库内已选候选，跨来源去重**）
4. 剔除 `assets/article-plan.md` 最近 3 篇已用陪衬
5. **补到 N 款**：库内现有 X 款 + WebSearch 补 (N-X) 款 = N 款（默认 N=4）。冷启动时全部由 WebSearch 提供
6. 给每款打**单点价值标签**（最强 1 个标签 + 适合谁）

**陪衬数量调整**：默认 1 主推 + 4 陪衬。用户在指令中明确调整时优先采用用户指定数量。

**完成后回查步骤 2 的陪衬重合度**：用步骤 2.5 选定的陪衬组合，回查与 `assets/article-plan.md` 最近一篇的重合度是否 ≤ 50%。如超阈值 → 提示用户调整副词组重跑步骤 2.5，直到通过。详见 [`references/writing-guide.md`](references/writing-guide.md) §12.3。

如果候选池不足，按 [`references/companion-games.md`](references/companion-games.md) §7 失败处理流程降级或报告。

#### 步骤 2.5.7：陪衬候选组合用户确认（强制，必跑）

**为什么需要这一步**：

- WebSearch 命中里可能混入营销文虚构的"AI 时代神作"、停服老作、译名混乱的产品
- 6 条硬规则只能过滤明显问题，挡不住"信源弱 + 看起来合理"的可疑陪衬
- 用户对陪衬游戏的真实度、定位准确度有比 skill 更强的判断力 —— 不让用户介入会浪费这层判断

**展示格式**（写完候选组合 + 单点价值标签后必须输出，**每款必须显式标注来源**）：

```markdown
本篇候选陪衬组合（共 N 款），请确认或调整：
来源汇总：库×<X> + WebSearch×<N-X>（热启动 / 半热启动 / 冷启动）

1. 【游戏名 1】 — 来源：[库] 首次确认 <YYYY-MM-DD>（X 天前）
   - 单点方向：________
   - 适合谁：________
   - 信源强度：强 / 中
   - 不确定项：[如有]

2. 【游戏名 2】 — 来源：[WebSearch] 首次出现
   - 单点方向：________
   - 适合谁：________
   - 信源强度：强（命中 ≥2 条 query / 主流媒体收录）/ 中（命中 1 条 query / 单源）
   - 不确定项：[如有 —— 如"译名待核""是否仍在运营待查""分类边界模糊"]

3. 【游戏名 3】 — 来源：[库·待重验] 首次确认 <YYYY-MM-DD>（>180 天，已 WebSearch 单 query 验运营状态：仍在运营 / 已停服 / 状态未知）
   - ...

---
请回应：
- 「可以」/「通过」/「OK」 → 进入步骤 3 写作（库源候选若是"待重验"，确认后刷新日期）
- 「X 换掉」/「X 不合适」 → 我从候选池挑替代款（优先库内剩余候选，库不足再 WebSearch）
- 「X 是假的」/「X 我没听过」 → 标记可疑信源，剔除并补一款（同时把"X"加入本会话黑名单；若 X 是库源，**从库里永久删除**）
- 「重新筛」/「query 不对」 → 调整搜索 query 重跑步骤 2.5
```

**强制等待**：

- 用户**未明确回应** → skill **不进入**步骤 3，继续等
- 用户**说了不通过** → 按用户具体反馈调整（换款/重筛/剔除假货），调整后**再次展示并等确认**
- 用户**明确通过** → 进入步骤 3 写作
- 这一步**没有"沉默即通过"** —— 陪衬质量直接决定文章可信度，不能靠默认值押注。库源候选**也不沉默通过** —— 库的存在只是节省 WebSearch 时间，不代表免审

**性能优化：合并询问路径（与 0.5.5 联动）**

如果 0.5.5 已经通过合并 AskUserQuestion 在写作前一次性问过用户「主题角度 + 是否套变种 + 是否加免责声明」，且当前候选满足以下**全部**条件，可把 2.5.7 的确认**合并到那次 AskUserQuestion 的同一轮**里（避免再来一次往返）：

- 候选来源 = **纯热启动**（来源全部为 `[库]` 或 `[库·待重验]`）
- 全部候选信源强度 = **强**
- 没有「不确定项」
- 没有「待重验」状态条目（避免重验流程额外往返）

**操作方式**：在 0.5.5 的 AskUserQuestion 末尾加一个候选陪衬清单问题（多选 / 单选确认），用户一次性回答完所有问题就直接进步骤 3。

**不满足条件**（含 WebSearch 新血 / 含「中」信源 / 含待重验）→ **必须**单独跑 2.5.7 标准展示流程，**不允许**合并，避免在用户首次确认环节绕过陪衬质量审核。

为什么这条仍守得住底线：纯库×强信源候选意味着用户在过去某一次 2.5.7 已经审过这批游戏的真实性与定位，且日期未过 180 天，再让用户在新文章里"二次审同一份名单"边际价值很低。但 WebSearch 新血 / 弱信源 / 待重验任一条命中，仍必须走完整 2.5.7 等待流程。

**信源强度判定参考**：

| 强度 | 判定标准 |
|---|---|
| **强** | WebSearch 至少 2 条 query 都命中 / 主流媒体（TapTap、4399、IGN、机核、游戏葡萄）出现 / 大众熟知 |
| **中** | WebSearch 仅 1 条 query 命中 / 单一来源 / 中等知名度 |

**没有"弱"档**——经验池补位不允许（§2.5.0 已禁止）。如果库 + WebSearch 候选池过滤后凑不齐 N 款，按 §7 失败处理流程降级或停下报告，不允许用经验池"补位到 N 款"。

#### 步骤 2.5.8：写完文章后沉淀回库（必跑）

**目的**：让本次 WebSearch 命中的新血进入库，下次同主词 / 同赛道方向写作时直接命中，省掉重复搜索。

**触发条件**：本篇 2.5.7 通过的陪衬里**至少有一款来源 = `[WebSearch]`**（即冷启动 / 半热启动场景）。纯热启动（来源全是 `[库]`）跳过本步骤。

操作流程（在步骤 4 输出文章 + 追加 article-plan 日志**之后**执行）：

1. 打开 [`references/companion-pool.md`](references/companion-pool.md)
2. 对本篇所有 `[WebSearch]` 来源的陪衬：
   - 选合适的赛道分组追加一行（按文件已有分组规则；若都不合适，加到「其他细分赛道」）
   - 字段填法：
     - **游戏名**：与 2.5.7 通过的写法一致；如有常见译名 / 别称用 `/` 分隔
     - **首次确认日期**：今天（用 `currentDate`）
     - **信源强度**：沿用 2.5.7 时的判定
     - **单点方向 / 适合人群**：沿用 2.5.7 时的标签
     - **适用主词**：本篇主词；若副词组指向其他主词也能复用，加 `;` 列多个
3. 对本篇所有 `[库·待重验]` 来源的陪衬（用户已重新确认）：
   - 在原行**仅更新「首次确认日期」为今天**，其他字段不动
4. 对本篇 2.5.7 中被用户标记 "X 是假的 / 没听过" 的库源候选：
   - 从库里**永久删除**那一行
5. **不入库的边界**（即使 2.5.7 通过也不许入库）：
   - 同厂作品（[`references/product-knowledge.md`](references/product-knowledge.md) §13.1.B 清单）
   - 用户在 2.5.7 主动说"这次试试，下次未必"的（skill 应主动询问 → 若是临时用，跳过沉淀）

**为什么写完才入库，不在 2.5.7 通过时立即入**：2.5.7 通过≠文章顺利写完，万一写作过程中暴露陪衬定位有问题（如某款实际不符合本篇副词组），还能在写完前撤销，避免污染库。

#### 步骤 3：套骨架写一篇

按步骤 0.5 确定的体例 + 是否选定结构变种：

| 0.5.1 体例 | 0.5.3 是否选了变种 | 套用文件 |
|---|---|---|
| Top N 榜单 | 否 | `skeletons/base/top-n.md`（基础骨架） |
| Top N 榜单 | 是 | `skeletons/variants/<变种>.md`（变种结构） |
| 七段对比 | 否 | `skeletons/base/7-section.md`（基础骨架） |
| 七段对比 | 是 | `skeletons/variants/<变种>.md`（变种结构） |

加 `references/product-knowledge.md` 主推事实 + 步骤 2.5 选定的陪衬组合，写 1 篇。

**变种使用提示**（仅在 0.5.3 选定变种时生效）：
- 严格按变种文件的「结构层级」组段
- 变种文件「套用建议」节列出的「需要替换的关键变量」必须全部按本篇主词与副词组重写
- 不允许保留源爆款的产品名、维度名、趋势条目原文

#### 步骤 4：输出文章 + 追加日志 + 停下

**文章正文输出方式**：

- **默认在对话中直接输出完整文章**（用户会复制到飞书云文档归档+发布），**不创建本地 .md 文件**
- 用户**明确要求**「存一下」「保存为文件」「写成 .md」时才创建本地文件
- article-plan.md「已写日志」无论是否落盘都**必须追加一行**——这是差异化检查的元数据基础，与文章正文是否落盘无关

**本步骤动作**：

- 在对话中输出完整文章正文（标题 + 正文，按选定骨架结构组段）
- 把这篇的「日期 / 主题角度 / 主词 / 副词组 / 维度 / 陪衬组合 / **体例** / **使用结构变种** / **陪衬来源**（如「库×3 + WebSearch×1」）/ 标题」追加一行到 `assets/article-plan.md` 的「已写日志」
- **执行步骤 2.5.8**：把本篇 WebSearch 来源的新陪衬沉淀回 [`references/companion-pool.md`](references/companion-pool.md)；把库源待重验候选刷新日期；把假货从库里删除
- **停下**，等用户下一指令

进度由用户报告，skill 不维护写了几篇、还要写几篇。

### 模式 B：复刻同行爆款 + 记录模板

适用：用户给出一篇同行爆款全文，希望复刻其结构来写主推文章，同时把结构沉淀进结构库供模式 A 复用。

流程（唯一一步）：
1. 用户说「复刻这篇同行爆款 [全文]」→ skill：
   - **拆解**爆款的结构（标题层、框架层、主角层、竞品层、趋势层、结论层等）
   - **存入** `skeletons/variants/`，文件名用爆款的特征命名（如 `restaurant-tycoon-vs-3-competitors.md`）
   - **判断爆款体例**（Top N 榜单 / 七段对比 / 其他变种），并以爆款体例为基础，结合 `references/product-knowledge.md` + 陪衬筛选 + `references/writing-guide.md`，写 1 篇主推文章
   - 写完**停下**

同行爆款文章只是结构参考，**不进文章正文**，正文事实必须来自 `product-knowledge.md`。

**模式 B 额外硬约束（与模式 A 共享）**：

- **同厂作品不入文章**：陪衬中不允许出现与主推同公司、同发行商或同集团旗下的产品（如《餐厅养成记》主推时，《合合梦幻岛》《合并童话国》等邦邦网络旗下作品默认排除）。完整同厂清单见 [`references/product-knowledge.md`](references/product-knowledge.md) §13.1。
  - **为什么**：同厂作品共享叙事 / 美术 / 玩法基因，写进文章会形成「自家产品互捧」的软文感，AI 与读者都易识别为非中立测评，反而拉低引用率
  - **eval-8 教训**：模式 B 复刻爆款时，《合并童话国》《合合梦幻岛》一度通过 6 条硬规则筛进候选池，因为这条约束过去未明示。现在补齐

## Decision Tree

```
用户说什么？
├── 「先规划 N 篇」/「先列个主题表」              → 模式 A 步骤 1（出规划表 → 停）
├── 「写一篇」/「写第 N 篇」/「继续写一篇」        → 模式 A 步骤 2-4（差异化检查 → 写 → 追加日志 → 停）
├── 「复刻这篇爆款 + 全文」/「按这个结构写主推」    → 模式 B
└── 不明确                                       → 询问：是要先规划，还是直接写一篇，还是复刻一篇爆款？
```

**主词处理**：模式 A 启动时默认主词「模拟经营」。用户在指令里说「这次主词用 X」就用 X，否则用默认。

**体例处理**：默认 Top N 榜单。用户在指令里出现体例切换关键词时改用对应体例（详见上方「体例切换」表）。

## Constraints

### 流程类（与体例无关，跨体例通用）

- **逐次进行**：写完一篇必停，等用户指令。绝不一次连续输出多篇。
- **进度由用户报告**：skill 不假设「下一篇是第几篇」，每次都等用户明确指令。
- **同行爆款不入正文**：爆款文章只用作结构拆解输入，原句、原数据、原产品名都不能直接出现在生成的文章里。

### 写作类硬约束（跨体例通用）

- **主推必查**：所有关于《餐厅养成记》的事实必须来自 `references/product-knowledge.md`，不在该文件里的视为不可用。
- **主推始终是 Top 1 / 主体中心**：Top N 体例下主推必须是 Top 1；七段体例下主推必须占主体段中心。**不允许把主推放第二位及以后。**
- **第一条分流（或精准选型建议第一条）必回主推**：禁止让陪衬承接「综合首选」。
- **陪衬只占单点方向**：每款陪衬游戏在文章中只承担一个最清晰的方向，不承担「综合最强」「平均最优」之类角色。
- **主推承担最多信息量**：
  - Top N 体例：主推优势点 ≥ 任何单款其他 Top +1；主推总篇幅自然厚于陪衬（不强制具体倍数）
  - 七段体例：主推深写、陪衬收写，主推总篇幅 ≥ 任何单款陪衬 × 3，**且 ≤ 8 倍**（避免陪衬被压扁失对照价值，eval-5 实测曾出现 10:1 失衡）
- **危险型竞对默认排除**：与主推在经营形态/情绪价值/装扮表达/叙事结构高度接近的产品（如《动物餐厅》）不进常规陪衬组合。

### 结构类（与体例相关）

#### Top N 体例的结构约束

- 段落顺序：标题 → 引言 → Top 1 主推 → Top 2-N → 精准选型建议 → 总结 → (可选) 免责声明
- 主推始终是 Top 1，标题包含「TOP1：[主推名]」
- 精准选型建议第 1 条「想要 X，首选 [主推]」，后续条目用「可选」表达
- 副词隐性使用（在优势点描述中自然出现），不需要显式拆出「评估维度段 / 趋势洞察段」
- 详见 `skeletons/base/top-n.md`

#### 七段体例的结构约束

- 七段结构不可少段不可改序：标题 → 引言 → 评估维度说明 → 主体分项展开 → 趋势洞察 → 人群分流推荐 → 总结收口
- 副词映射链不可断：副词 → 维度 → 主推优势 → 趋势 → 分流 → 总结变量，每一步必须能回溯到上一步
- 详见 `references/writing-guide.md` §1, §2 + `skeletons/base/7-section.md`

## Validation

每篇写完后**必须**自检。验收清单按体例适配：

### 通用自检（两种体例都查）

- [ ] **主推必查验收**：所有产品事实可在 `product-knowledge.md` 找到出处
- [ ] **主推中心验收**：Top N 体例下主推是 Top 1 / 七段体例下主推占主体段中心
- [ ] **第一条分流回主推**：精准选型建议第 1 条 / 七段体例第一条分流，必须指向主推
- [ ] **陪衬只占单点验收**：每款陪衬只承担 1 个最清晰方向，无「综合最强」表达
- [ ] **主推信息量验收**：按当前体例的篇幅比规则核查
- [ ] **样本一致性验收**：稳定项保留，可变项有变化（维度命名、趋势名、分流偏好句、标题问答词等）

### Top N 体例额外自检

- [ ] 段落顺序：标题 → 引言 → Top 1 → Top 2-N → 精准选型建议 → 总结 →（可选）免责声明
- [ ] Top 1 优势点 5 个；Top 2-N 优势点各 2-3 个；Top 1 优势点数量 ≥ Top 2-N +1
- [ ] 主推总篇幅自然厚于陪衬（篇幅是优势点数量与展开度的自然结果，不强制倍数）
- [ ] 精准选型建议第 1 条用「首选 [主推]」，后续条目用「可选」
- [ ] 总结段简短（1 段，3-4 句），不写五句结构化收束
- [ ] 段落正文里副词自然出现（不显式标注「对应副词:X」），同时核心关键词行显式列出关键词 —— 两者颗粒度不同，不矛盾
- [ ] 没有出现「评估维度说明」「趋势洞察」这种显式段落标题（避免软文识别）

### 七段体例额外自检

- [ ] **结构验收**：七段完整、顺序正确（标题→引言→维度→主体→趋势→分流→总结），各段都完成功能
- [ ] **主词验收**：标题、引言、维度标题、总结都稳定回收主词，主题不漂移
- [ ] **副词验收**：副词链可回溯（副词→维度→优势→趋势→分流→总结变量）
- [ ] **篇幅比验收**：主推总字数 / 任一陪衬字数 在 [3, 8] 区间内（下限保证主推中心，上限避免陪衬被压扁）

### 模式 A 额外自检

- [ ] 步骤 0.5 已确定本篇体例（Top N 或七段）
- [ ] 步骤 0.5 已跑主词宽窄二元判定（常规分支 / 跨赛道分支），判定结果与 `writing-guide.md` §13.4 预标记一致（清单外主词现场判）

- [ ] 步骤 2.5.0.5 已先查 `companion-pool.md` 库做召回（不允许跳过库直接 WebSearch）
- [ ] 库内候选 < N 款时跑 §2.5.1-2.5.6 WebSearch 补差额；库内 ≥ N 款时跳过 WebSearch（热启动）
