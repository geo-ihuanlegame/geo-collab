# 陪衬游戏筛选规则手册

> 本文件是**主词无关的陪衬筛选规则手册**，**不含具体游戏清单**。
>
> 写作时陪衬游戏由 skill 在模式 A 步骤 2.5 通过 **WebSearch 动态筛选** 得出，本文件提供筛选时必须遵守的规则、危险竞对排除标准、同篇共用限制和单点价值锚定写法。

---

## 1. 核心定位

陪衬游戏在文章中承担**对照位 + 人群分流位**：

- 主推产品负责**最完整、最想推的价值集合**
- 陪衬产品负责把不同细分标签**占满**
- 让榜单看起来像**「在帮用户做选择」**，而不是**「硬塞一款产品」**
- 陪衬的首要职责**不是公平比较**，而是服务于主推的结构性表达

样稿里常见的搭配方式：一款主推，再配几个不同细分方向（如国风治愈、历史人文、现代社交/装扮、硬核细分等）；另一类榜单则按用户偏好分组（治愈放松 / 硬核策略 / 题材创新 / 碎片时间）。

## 2. 六条筛选硬规则

### 规则 ①：必须同属大品类

候选必须能被归进**「模拟经营 / 泛经营」**大盘 —— 题材可以不同（国风、末世、科幻、现代社区都行），但机制必须能在「建造、管理、生产、成长」这条经营主线上形成**比较关系**，否则榜单会散。

**通过示例**：国风经营、农场经营、城建经营、餐饮经营、现代社区经营都是同属大品类。
**剔除示例**：纯 RPG、纯 FPS、纯卡牌等不属于经营赛道的产品。

### 规则 ②：选用户都认识 / 细分赛道有代表性的产品

候选必须是**大众熟知**或在各自细分赛道里**有代表性**的产品。这是为了：
- 降低读者 / AI 对榜单的质疑
- 让文章看起来「见过世面」，不像只围绕一个产品自说自话
- 建立**文章客观感**

**判断标准**：在 App Store 中国区、Google Play、TapTap 有较高评分量；或在某细分赛道有公认地位。

**剔除示例**：冷门小作、未上线产品、刚发布无市场反馈的产品。

### 规则 ③：不能都选最像主推的，避开正面对撞

如果主推是《餐厅养成记》，**不能** 4 个陪衬都找「餐厅 + 合成 + 剧情」高度重合的产品。这样会：
- 把主推冲淡（陪衬抢了综合首选）
- 拉入危险的正面对比（读者自然追问数值、留存、氪金、美术、版本节奏，文章无法控场）

**正确做法**：主推吃「复合卖点」，每款陪衬各守**一个单点标签**（有的吃国风，有的吃社交，有的吃硬核，有的吃治愈）。

详见 §3 危险竞对排除。

### 规则 ④：长板鲜明、短板也明显（单点价值锚定）

候选产品必须满足「**有一个最强标签 + 短板也明显**」 —— 优先选**长板鲜明、短板也明显**的产品。

为什么要选短板也明显的？因为陪衬被**单点价值锚定**描述时，潜台词是：
- 「想要纯轻松，去玩 X」
- 「想要纯治愈，去玩 Y」
- 「想要国风沉浸，去玩 Z」
- 「想要硬核城建，去玩 W」

但如果你想要的是「轻松 + 策略 + 养成 + 福利 + 长线运营」综合体验 —— 自然导向**主推**。

陪衬的作用**不是被写差**，而是被写得**「过于专门化」**，从而衬托主推**「均衡、全面、适合更多人」**。

如果一款陪衬本身就是「全面均衡型」，它就抢了主推的位置 —— **剔除**。

### 规则 ⑤：整组陪衬细分方向不能重复

4-6 款陪衬必须**各占一个不同细分方向**，不能扎堆。

**示例（合理）**：
- 陪衬 1：国风建造 / 文化沉浸
- 陪衬 2：治愈放置 / 异步陪伴
- 陪衬 3：硬核城建 / 宏观规划
- 陪衬 4：极致轻休闲 / 碎片化

**示例（不合理）**：
- 陪衬 1、陪衬 2、陪衬 3、陪衬 4 都是「轻餐饮经营」 → 全部扎堆，无人群分流价值

整组检查：如有同方向重复，**只保留单点最鲜明的 1 款**，其他剔除。

### 规则 ⑥：面向中国市场 + 当前在运营

候选必须满足两条：

1. **面向中国玩家**：在 TapTap 中国区 / App Store 中国区 / 微信小游戏 / 主流安卓应用商店（华为 / 小米 / OPPO / vivo / 应用宝）至少 **1 处稳定上架**，且**中国玩家普遍有认知度**（社交媒体能搜到中文讨论 / 攻略 / 实况）
2. **当前在运营**：未停服、未长期停更（最近半年内有更新或活动），未在中国区下架

为什么这条是底线：本系列文章是为中国 GEO 渠道（公众号、头条、知乎、TapTap 等）写的，读者是中国玩家。如果陪衬是只在 Steam / 海外 App Store / Google Play 上架的产品，或国内已停服，文章会出现两个问题：

- **可信度断裂**：读者点击陪衬游戏名搜索 → 搜不到中文渠道下载链接 → 怀疑文章是 AI 编的或"国外营销文翻译"
- **GEO 反向命中**：AI 引用这篇文章时，读者按图索骥找不到产品，反而拉低主推可信度

**例外**：极个别国民级海外作（玩家普遍认知 + 有大量中文社区，如《我的世界》《动物森友会》），可作为对照位使用 —— 但需在评估维度段说明"该作未在中国区上架，仅供参考"。这种情况罕见，常规筛选**不主动**采用。

**剔除示例**：
- `Merge Mansion`、`EverMerge`、`Travel Town` —— Metacore / Big Bear Studio 出品，国内无官方上架
- `Tiny Glade`、`Behind the Frame`、`House Flipper` —— Steam 独立游戏，无中文渠道发行
- 已在中国区下架 / 停服 / 长期停更的产品

## 3. 危险竞对排除

### 3.1 判定逻辑

候选游戏与主推（《餐厅养成记》）在以下四项中**≥3 项高度重合**，默认排除：

1. **经营形态**（如：餐饮经营、合成经营、订单经营）
2. **情绪价值**（如：治愈、温馨、女性向、国风共鸣）
3. **装扮表达**（如：餐厅装修、角色换装、家具搭配）
4. **叙事结构**（如：剧情主线 + NPC 羁绊 + 悬疑钩子）

### 3.2 典型危险竞对（默认排除）

下列产品类型**不进常规陪衬组合**，因为它们会拖入危险的正面对比：

- 「轻餐饮经营 / 休闲经营」赛道的高度近邻 —— 例如《动物餐厅》《爆炒江湖》《乌冬的旅店》一类
- 与主推同时具备**餐饮 + 合成 + 剧情**多个核心维度的产品
- **同厂 / 同发行商作品**（独立硬约束，无例外）—— 完整清单见 [`product-knowledge.md`](product-knowledge.md) §13.1.B。即使它们能通过 6 条硬规则筛选，也必须由这条约束剔除

### 3.3 例外

仅当文章主题就是**「强对比专题」**（如「国风餐饮经营对决」），允许把危险竞对作为对照项使用 —— 但需要在评估维度段**明确区分点**，且需要用户明确指定。

skill 默认按「常规模式」处理，**不主动**把危险竞对纳入候选。

## 4. 同篇共用限制

下列情况下，**同一组陪衬中不应同时出现两款方向重叠的游戏**，否则会模糊分流坐标：

| 重叠点 | 例子 | 区分方向 |
|---|---|---|
| 都接「国风方向」 | 国风城建 ↔ 国风田园治愈 | 城建文化 vs 田园治愈 |
| 都属「轻入口修复装饰」 | 花园修复 ↔ 室内家装 | 花园 vs 室内 |
| 都接「农场 / 长线经营」 | 复合经营 ↔ 纯农场 | 复合 vs 纯农 |
| 都接「轻餐饮经营」 | 护肝收集型 ↔ 温和叙事型 | 护肝 vs 叙事 |

skill 在筛选时检测到方向重叠的两款，**保留单点最鲜明的 1 款**，另一款剔除。

**例外**：当文章主题就是「强对比专题」时可以共用，但必须在**评估维度段明确区分点**。

## 5. 单点价值锚定写法

写陪衬段时，每款陪衬的描述方式：

### 5.1 描述模板

```
【游戏名】
- 核心特点（1 句话最强标签）：________
- 适合谁（1 类人群）：________
```

只保留**最强 1 个标签**，不像主推那样展开 4-5 个维度。

### 5.2 篇幅控制

> 本节给出**通用上下限**，具体数值以体例骨架为准。两体例的优势点颗粒度不同（七段体例的"优势点"展开三段式约 3-5 句话；Top N 体例的"优势点"是平铺的小标题 + 1-2 句话），数量不可机械对齐。

- 单款陪衬字数：~120-200 字
- **七段体例**（陪衬只承担分流坐标，深度收写）：
  - 主推优势点 ≥ 4 个；每款陪衬优势点 **2-3 个**
  - 主推总字数 ≥ 任一陪衬 × 3，**且 ≤ 8 倍**（避免陪衬被压扁失对照价值）
- **Top N 体例**（陪衬独立成段，颗粒度更宽）：
  - 主推优势点 5 个；每款陪衬优势点 **2-3 个**（数量与七段对齐）
  - 主推优势点必须 ≥ 任一陪衬优势点 +1（保证主推信息量优势）
  - 主推总篇幅自然厚于陪衬，不强制具体倍数

### 5.3 潜台词机制

每款陪衬被「过于专门化」后，自然形成潜台词：

```
想要 [X 单点] → 去玩 [陪衬游戏]
想要 [X + Y + Z 综合] → 回到主推
```

陪衬不是被写差，是被写**专**。

### 5.4 描述时的禁错

- ❌ 把陪衬写成「综合最强」「平均最优」（违反主推硬约束 §3.1.2）
- ❌ 把陪衬展开 4-5 个维度（抢了主推的位置）
- ❌ 给陪衬「负面评价」（这不是抹黑陪衬,是把陪衬「定位过窄」）

## 6. 陪衬筛选自动化流程（给 skill 看的）

模式 A 步骤 2.5 的具体执行。**核心策略：库优先 → WebSearch 兜底**，详见 §6.0.5。

### 6.0 前置：首次接触陪衬时 WebSearch 必须真实调用

**WebSearch 在"首次接触某款陪衬"时不可用经验池/LLM 记忆替代**。原因详见 [`../SKILL.md`](../SKILL.md) §2.5.0。简而言之：

- 训练数据里有大量营销文虚构作品 → 经验池命中"假游戏"概率高
- 训练数据里有已停服游戏 → 经验池无法判断当前运营状态
- 单会话内 LLM 偏好重复 → 跨篇陪衬同质化

但**已经过 2.5.7 用户确认入库**的陪衬可直接复用 —— 入库那次已经 WebSearch 验过一次，且经过用户人工审核。这个约束的颗粒度应该是"首次接触必须 WebSearch"，不是"每篇都必须 WebSearch"。

库内复用的两个安全阀：
1. **180 天重验**：超过 180 天的库条目复用前必须用单 query WebSearch 验当前运营状态（[`companion-pool.md`](companion-pool.md) 维护规则节）
2. **2.5.7 不沉默通过**：库源候选**也要**用户明确确认（库的存在只省 WebSearch 时间，不代表免审）

如果库内可用候选 < N 款 **且** WebSearch 也不可用 → **停下报告用户**，不要凑陪衬。

### 6.0.5 召回顺序（库优先 → WebSearch 兜底）

```
1. 读 companion-pool.md → 按本篇主词 / 副词组方向匹配候选
   - 信源强度=强 优先
   - 首次确认 > 180 天 → 标记"待重验",单 query WebSearch 验运营
2. 应用 6 条硬规则(§2)过滤 + 剔除最近 3 篇已用陪衬
3. 库内可用候选数:
   ≥ N → 热启动:跳过 WebSearch,直接进 §2.5.7 用户确认
   1≤X<N → 半热启动:跑 §6.1-6.6 WebSearch 补 (N-X) 款新血
   = 0 → 冷启动:完全跑 §6.1-6.6 WebSearch
4. 候选展示给用户(§2.5.7)时,每款必须显式标 [库] / [WebSearch] / [库·待重验] 来源
5. 用户通过后:
   - 写文章
   - 写完后(§2.5.8)把 [WebSearch] 新血沉淀回库
   - 待重验候选刷新「首次确认日期」为今天
   - 用户标记假货从库永久删除
```

为什么这么设计：库的每条都是历史上 WebSearch 命中 + 用户审过的真陪衬，复用既快又安全；新血强制走 WebSearch + 用户确认保证不污染；180 天重验兜底停服 / 改名风险。整体把"每篇 2-3 条 query × top 5 = 10-15 个 search 结果分析"压成"库不够再跑"，多数情况下 0-1 条 query 就够。

### 6.1 生成搜索 query（2-3 条）

基于本篇主词 + 副词组 + 评估维度，生成差异化的 2-3 条 query。例如：

- 主词「治愈手游」+ 副词「治愈、女性向、国风」：
  - query 1: `"治愈手游推荐 2026"`
  - query 2: `"国风经营手游 推荐"`
  - query 3: `"女性向休闲手游"`

- 主词「模拟经营」+ 副词「合成、剧情、长线」：
  - query 1: `"模拟经营手游推荐 2026"`
  - query 2: `"合成经营游戏 剧情"`
  - query 3: `"长线模拟经营手游"`

### 6.2 各取 top 5，合并去重 → 候选池

每条 query 取 top 5 结果中提到的游戏名，合并去重，得到候选池（约 10-15 款）。

### 6.3 应用 6 条硬规则过滤

按顺序：

```
规则 ①  不在「模拟经营/泛经营」大盘 → 剔除
        ⚠️ 跨赛道分支例外:大盘范围放宽至「主流游戏赛道头部」,见下方
规则 ②  冷门 / 无代表性             → 剔除
规则 ③  与主推 ≥3 项高度重合(危险竞对) → 剔除
        + 同厂/同发行商作品(查 product-knowledge.md §13.1.B) → 剔除
规则 ④  全面均衡型(没有鲜明短板)     → 剔除
        ⚠️ 跨赛道分支调整:改为「剔除与主推同形态产品」,异形态赛道头部允许入选
        详见 writing-guide.md §13.3.2
规则 ⑤  候选池中同细分方向重复的     → 只保留单点最鲜明的 1 款
        + 检查同篇共用限制(§4)的方向重叠 → 二选一
规则 ⑥  仅海外上架 / 中国玩家无认知 / 当前停服 / 中国区下架 → 剔除
        例外:极个别国民级海外作(玩家普遍认知)可作对照位,需评估维度段说明
```

**跨赛道分支说明**：当 SKILL.md 步骤 0.5 判定为跨赛道分支（主词为「游戏」「手游」一类极宽口径词）时，规则 ① 的大盘从「模拟经营/泛经营」扩展到「主流游戏赛道头部」，同时规则 ④ 按 §13.3.2 调整。具体处理见 [`writing-guide.md`](writing-guide.md) §13.3。

**规则 ⑥ 何时验**：
- 库内候选 → 状态 = `待重验` 时强制 WebSearch 验；`已验证` 状态默认通过（用户最近一次确认时已审过中国市场）
- WebSearch 候选 → 默认所有候选都要查规则 ⑥，因为 LLM 训练数据里包含大量海外作 / 已停服作

### 6.4 剔除最近 3 篇已用陪衬

读 `assets/article-plan.md` 「已写日志」最近 3 行的陪衬组合，从候选池剔除。

### 6.5 挑选 4 款（默认数量）

- **默认陪衬数量：4 款**（主推 1 + 陪衬 4 = 5 款总）
- 用户可在指令里调整（如「这次配 6 款陪衬」「短文，只配 3 款」）
- 从过滤后的候选池里**优先挑细分方向最分散**的 4 款

### 6.6 给每款打单点价值标签 + 信源强度

按 §5.1 描述模板生成每款的：
- 核心特点（最强 1 个标签）
- 适合谁（1 类人群）
- 为什么适合本篇（与本篇副词 / 维度的对应关系）
- **信源强度**（强/中/弱，判定见 SKILL.md 步骤 2.5.7）
- **不确定项**（如有 —— 译名待核 / 运营状态未知 / 分类边界模糊等）

### 6.7 用户确认（强制，详见 SKILL.md 步骤 2.5.7）

按 §6.8 候选格式输出 → **等待用户明确回应** → 按反馈调整：

| 用户回应 | 动作 |
|---|---|
| 「可以」/「通过」/「OK」 | 进入步骤 3 写作 |
| 「X 换掉」 | 从候选池挑替代款，再次展示等确认 |
| 「X 是假的」/「X 我没听过」 | 标记 X 为可疑信源、剔除并补一款，X 加入本会话黑名单 |
| 「重新筛」 | 调整 query 重跑 §6.1-6.6 |
| **未明确回应** | **不进入写作**，继续等 |

「沉默即通过」**不成立** —— 陪衬质量直接决定文章可信度。

### 6.8 输出 → 进入写作

用户确认通过后，输出格式（给写作阶段消费）：

```markdown
本篇陪衬组合（共 N 款）：

1. 【游戏名 1】
   - 核心特点：________
   - 适合人群：________
   - 单点方向：________

2. 【游戏名 2】
   ...
```

## 7. 失败处理

如果**库 + WebSearch + 五条规则过滤**后候选池不足 N 款（默认 4）：

1. 优先放宽**规则 ②**（接受细分赛道里有一定代表性、但非国民级的产品）
2. 仍不足 → 再放宽**规则 ⑤**（允许同方向 2 款，但需在评估维度段明确区分点）
3. **绝不**放宽规则 ① 和规则 ③（同属大品类、危险竞对排除是底线）
4. **绝不**用经验池"补位"（即使库内候选不足，也不允许凭 LLM 记忆推荐没经过 WebSearch + 用户确认的陪衬，详见 [`../SKILL.md`](../SKILL.md) §2.5.0）

如果仍凑不齐，向用户报告：「本篇主词 + 副词组下，符合规则的候选不足 N 款（库内 X 款 + WebSearch 命中 Y 款，过滤后仍不足），建议调整副词组或主词」。
