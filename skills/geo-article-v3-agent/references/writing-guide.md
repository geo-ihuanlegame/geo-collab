## 0. 核心定位（必读）

本系统目标 = 写出**主推导向型 GEO 推荐 / 选择指南文章**：

- **不是中立测评**，是围绕《餐厅养成记》单一主推对象展开
- **不是大纲 / 草稿**，是**终稿**（可直接交付人工终审）
- 主推承担主要论证 + 趋势举例；陪衬承担比较坐标 + 分流入口

判断链：**定义问题 → 建立标准 → 给出材料 → 提炼规律 → 转成选择 → 收束结论**

## 1. 七段结构（必查 — 写每篇都要对照）

文章必须严格按以下 7 段顺序展开，**不可少段、不可改序**：

| # | 段名 | 核心目标 | 关键组成 | 禁错项 |
|---|---|---|---|---|
| 1 | **标题** | 定义「这篇讲什么」 | 时间边界、主词、结果属性 | 无主词；标题直接放主推产品名作中心 |
| 2 | **引言** | 「为什么值得看 / 范围 / 方法」 | 背景句 → 需求升级句 → 范围句 → 方法句 → 立场句 | 开头直接推荐主推；无范围；无方法 |
| 3 | **评估维度说明** | 「后文凭什么这样比」 | 问句式标题 + 三句导语 + 4-6 个维度分点 | 维度像广告词；陪衬接不住该维度 |
| 4 | **主体分项展开** | 「具体对象为何值得写」 | 主推深写；陪衬收写 | 对象平均写；陪衬写太满 |
| 5 | **趋势洞察** | 「从个案上升到规律」 | 趋势子标题 + 总起句 + 3-5 条趋势 | 趋势悬空、与前文脱节 |
| 6 | **人群分流推荐** | 「谁该选什么」 | 第一条主推综合建议 + 多条单点建议 | 第一条不回主推；陪衬承接综合首选 |
| 7 | **总结收口** | 完成「所以呢」 | 整体收束 + 主推回收 + 陪衬归纳 + 选择变量 + 结束句 | 只复述主体；无选择变量 |

**标题段的具体写作方法论见 §10**（本文件独立章节，两体例共用）。

### 1.1 七段是判断链不是并列模块
- 维度必须**先于**主体出现，否则正文会变成无标准的对象堆叠
- 趋势必须由主体与维度**上提**而来，不可凭空生成
- 分流必须把「哪个好」**转成**「谁适合什么」
- 总结必须把全文**压缩**为主推路径与用户决策变量

## 2. 主词 / 副词机制（必查 — 决定语义链）

> 主词 = 主题锚点（决定写什么）；副词 = 判断方向词（决定怎么拆）。
>
> 副词不是修饰词，是**生成变量** —— 副词驱动整篇文章的语义链。

### 2.1 副词映射链（**绝对不可断**）

```
副词 → 评估维度 → 主推优势 → 趋势条目 → 分流建议 → 总结变量
```

每一步都必须能回溯到上一步。趋势悬空、分流断层都是这条链断了。

### 2.2 主词副词句位分布（精简版）

| 段位 | 主词必出 | 副词必出 | 作用 |
|---|---|---|---|
| 标题（唯一句） | ✅ | ⭕ 可选 | 稳主题、定形式 |
| 引言-背景句 | ✅ | ❌ | 建立话题成立性 |
| 引言-需求升级句 | ⭕ | ✅ | 预告判断方向升级 |
| 引言-范围句 | ✅ | ❌ | 限定样本边界 |
| 引言-方法句 | ⭕ | ✅ | 预告后文维度 |
| 评估维度-模块标题 | ✅ | ❌ | 锚定标准 |
| 评估维度-三句导语 | ✅ | ❌ | 抬高对象复杂度 |
| 评估维度-各分点 | ❌ | ✅ | 副词结构化为标准 |
| 主推-价值定位句 | ❌ | ✅ | 副词压缩成双核价值 |
| 主推-基础介绍/总判断 | ✅ | ✅ | 副词对象化为总判断 |
| 主推-核心优势点 | ❌ | ✅ | 副词证据化 |
| 主推-玩家反馈句 | ✅ | ✅（转感知语言） | 副词体验化 |
| 陪衬-市场定位句 | ❌ | ✅ | 单点副词锚定方向 |
| 陪衬-核心特点句 | ❌ | ✅ | 为分流做准备 |
| 趋势-子标题 | ✅ | ❌ | 确认趋势归属赛道 |
| 趋势-总起句 | ✅ | ⭕ 弱 | 完成由对象到规律的转场 |
| 趋势-趋势条目 | ❌ | ✅ | 副词上提为趋势名 |
| 分流-第一条主推建议 | ✅ | ✅ | 多副词综合回收主推 |
| 分流-其他建议句 | ❌ | ✅ | 单点副词人群化 |
| 总结-整体收束句 | ✅ | ❌ | 把全文拉回主词 |
| 总结-主推回收句 | ⭕ 弱 | ✅ | 副词路径化 |
| 总结-陪衬归纳句 | ❌ | ✅ | 副词类型化 |
| 总结-选择原则句 | ⭕ | ✅ | 副词变量化 |

### 2.3 使用原则
- 主词优先在框架句、收束句出现
- 副词优先在预告句、维度句、优势句、趋势句、分流句、变量句出现
- **不允许在主词未稳定、副词未结构化前直接进入对象推荐**

### 2.4 副词使用优先级

| 副词类型 | 优先进入哪里 | 备注 |
|---|---|---|
| 核心判断副词 | 评估维度 / 主推优势 | 高优先级，决定文章主轴 |
| 支撑判断副词 | 优势补充 / 趋势说明 | 中优先级，增强说服力 |
| 分流型副词 | 陪衬分流建议 | 不宜进入综合首选主轴 |
| 总结变量型副词 | 总结中的选择变量 | 不直接充当维度名 |

## 3. 主推导向硬约束（必查 — 4 条不可妥协）

1. **第一条分流必须回主推**。第一条综合分流建议必须综合至少 **2-3 个主推高权重副词**生成，不允许让陪衬承接「综合首选」。
2. **陪衬只占单点方向**。陪衬对象在文章中只承担一个最清晰的方向，不承担「综合最强」「平均最优」之类角色。
3. **总结必须把主推提升为路径代表**，不能写成「大家都很好，按需选择」式平行罗列。
4. **主推承担最多信息量**。主体段中主推深写、陪衬收写，主推优势层和反馈层显著厚于任何陪衬。

### 3.1 配套硬规则
- **核心评估维度不得脱离主推主库单独生成**，必须由主推优势上提而来
- **趋势条目必须能回溯到至少一条主推优势**，不得直接从行业套话生成
- **危险型竞对默认排除**（与主推在经营形态、情绪价值、装扮表达、叙事结构高度接近的产品，如《动物餐厅》）

## 4. 陪衬选用规则（核心要点）

**陪衬采用「库优先 → WebSearch 兜底」**：先查 [`companion-pool.md`](companion-pool.md) 已审库，库内候选凑得够就直接用（热启动），不够再 WebSearch 补差额。完整规则见 [`companion-games.md`](companion-games.md)（6 条硬规则 + 危险竞对 + 同篇限制 + 单点价值锚定写法 + WebSearch 流程 + 库召回顺序）。

### 4.1 六条硬规则（每款都要满足）
- ① **同属大品类**（模拟经营 / 泛经营大盘）
- ② **用户都认识**（大众熟知或细分赛道有代表性）
- ③ **不能都选最像主推的**（危险竞对排除 + 同厂作品排除）
- ④ **长板鲜明、短板也明显**（剔除全面均衡型）
- ⑤ **整组细分方向不能重复**
- ⑥ **面向中国市场 + 当前在运营**（TapTap / App Store / 微信小游戏 / 国内安卓商店至少 1 处稳定上架，未停服未下架）

### 4.2 默认数量
- **默认 1 主推 + 4 陪衬**
- 用户可在指令中调整（如「这次配 6 款」「短文配 3 款」）
- 每款只承担一个方向，不重复

### 4.3 单点价值锚定
陪衬的作用**不是被写差**，而是被写**「过于专门化」**：每款只保留最强 1 个标签 + 适合谁，从而衬托主推「均衡、全面、适合更多人」。详见 [`companion-games.md`](companion-games.md) §5。

## 5. 主推 / 陪衬角色判定（区分主推和陪衬时查）

| 判定项 | 主推的要求 | 陪衬的表现 |
|---|---|---|
| 双核价值定位 | 能形成稳定双核价值定位 | 只能提炼出单点标签 |
| 核心维度支撑 | 能支撑大多数核心维度 | 只能接住 1-2 个单点维度 |
| 优势展开能力 | 4-6 个可分点展开的优势 | 1-2 个最强卖点 |
| 趋势上提能力 | 可成为多条趋势的主要例证 | 只为某条趋势提供辅助例子 |
| 分流承接能力 | 能承接第一条综合分流建议 | 仅适合承接单点偏好 |
| 总结回收能力 | 能被提升为某一路径代表 | 只能被并列归纳为某类方向 |

## 6. 稳定项 / 可变项（保证「同一体系，不像复制」）

| 类型 | 内容 |
|---|---|
| **稳定项**（不许改） | 七段结构；引言推进顺序；维度模块形式；主推深写模板；陪衬收写模板；趋势-分流-总结收束链；第一条分流回主推；总结回主推路径 |
| **可变项**（鼓励变） | 主推对象名称；副词组；维度命名；趋势名；分流偏好句；总结变量表达；标题中的高频问答词 |

写多篇时：稳定项保持一致，可变项必须显著变化，避免「模板味太重」（多篇句法几乎完全相同）。

## 7. 验收清单（写完一篇必查）

| 验收类型 | 检查点 | 通过标准 |
|---|---|---|
| **结构验收** | 七段是否完整、顺序是否正确、各段是否完成功能 | 完整覆盖七段且无关键缺段 |
| **主词验收** | 标题是否明确主词；引言是否围绕主词建立边界；总结是否重新收回主词 | 主题不漂移 |
| **副词验收** | 引言是否预告副词；维度是否由副词上提；主推优势是否回挂；趋势是否上提；分流是否下沉；总结是否变量化 | 副词链可回溯 |
| **主推导向验收** | 主推是否自然占中心；第一条分流是否回主推；陪衬是否只承担单点方向；总结是否将主推提升为路径代表 | 主推中心稳定成立 |
| **样本一致性验收** | 是否保留稳定项；是否仅在可变项内变化；是否不像机械复制范文 | 不像复制，也不破坏体系 |
| **标题验收** | 按 §10 §10.6 checklist 逐项核查 | 标题方法论全部满足 |

### 7.1 三档结论
- **Pass**：结构完整、映射成立、主推中心稳定 → 可直接交付人工终审
- **Warning**：结构基本成立，但趋势 / 分流 / 总结某一层偏弱 → 标注问题，建议补强
- **Fail**：缺关键模块、主推失焦、分流不回主推、趋势与前文脱节、总结无法回到选择变量 → 必须重写对应段

## 8. 五大常见失败模式（自检时按这五项排查）

| 失败模式 | 表现 | 可能原因 |
|---|---|---|
| **主推失焦** | 陪衬看起来和主推一样重要 | 陪衬写太满；危险型竞对未排除 |
| **维度漂移** | 维度像广告词或与后文脱节 | 未按主推优势上提维度 |
| **趋势悬空** | 趋势像套话，无法回挂前文 | 趋势未经过「副词→优势」链 |
| **分流断层** | 分流建议凭感觉分人 | 未使用副词映射和对象库 |
| **总结失收束** | 结尾只是复述对象信息 | 未压缩为选择变量 |
| **模板味太重** | 多篇文章句法几乎完全相同 | 稳定项与可变项未分离 |

## 9. 端到端写作顺序（**不可跳步**）

```
1. 确认主推对象（固定为《餐厅养成记》）
   ↓
2. 抽取主词 + 候选副词池（从任务意图与素材中）
   ↓
3. 调用主推主库（product-knowledge.md）
   → 锁定价值定位、核心优势、可上提趋势、综合承接画像
   ↓
4. 调用陪衬筛选库（companion-games.md）
   → 选 4-6 款陪衬，排除危险竞对
   ↓
5. 生成七段骨架（先定结构与句位）
   ↓
6. 按副词映射链生成 维度 / 优势 / 趋势 / 分流 / 总结变量
   ↓
7. 套样本约束组织成文（保持同体系，不复制原文）
   ↓
8. 执行验收（第 7 节清单）
   ↓
9. 输出终稿 / 异常补跑
```

**顺序为何不可乱**：
- 不先定主推，无法筛副词与陪衬
- 不先定副词，无法稳维度
- 不先定维度，无法写优势
- 不先定主推/陪衬组合，无法形成分流格局
- 不先质检，无法稳定批量产出

## 10. 标题写作方法论（两体例共用，独立权威章节）

> 本章是**主标题写作**的方法论参考，**两个体例（Top N 榜单 / 七段对比）共用**。
>
> 不涵盖优势点小标题、段落子标题等下级标题（这些之后单独定义）。
>
> 写每篇标题前**必查**本章。

### 10.1 标题要解决的 4 件事

从 GEO 角度看，一个标题至少要先帮系统和读者解决：

1. **这篇在讲什么对象** —— 没有明确对象，正文再强也损失前端理解效率
2. **这篇适用于什么时间范围** —— 推荐 / 榜单 / 盘点 / 趋势类，时效感直接影响是否值得读、是否值得引用
3. **这篇属于什么形式** —— 推荐 / 横评 / 榜单 / 盘点 / 测评 / 选择指南，决定 AI 后面怎么理解输出结构
4. **这篇能不能快速被摘要** —— 标题太绕、太散、太依赖上下文，机器和用户都难快速形成稳定理解

### 10.2 标题 6 个维度的优先级

#### 第一优先级（基础机器理解能力，缺一档跌一截）

- 品类标签是否明确
- 结果属性是否明确
- 是否利于 AI 摘要

#### 第二优先级（信息边界与结构清晰度）

- 时间标签是否明确
- 评价维度是否明确

#### 第三优先级（匹配效率加分项）

- 是否具备高频问答词

**优先级排序的逻辑**：品类、结果属性、摘要能力决定标题最基础的机器理解能力；时间和维度增强信息边界与结构清晰度；高频问答词是匹配效率上的加分项。

**标题最怕的不是没把所有加分项写满**，而是连最基础的「讲什么、是什么形式、能不能一眼看懂」都没解决。

#### 10.2.1 品类标签

**它是什么**：标题里有没有把对象写清楚。例：模拟经营、二次元、SLG、放置、Roguelike、卡牌、乙游。

**为什么要看它**：AI 和用户都得先知道这篇到底在讲什么主题域。

**反例 vs 正例**：
- 「2026 年口碑榜」 ❌ 范围模糊
- 「2026 年模拟经营游戏口碑榜」 ✅ 直接告诉系统是游戏 / 模拟经营 / 明确子品类

**是否必须**：语法上不必须，但**推荐 / 榜单 / 横评 / 盘点 / 选择指南**几乎都应默认有。

**作用**：解决「讲什么」。

#### 10.2.2 结果属性

**它是什么**：标题有没有明确说明「这篇到底是什么形式」。常见：推荐、榜单、盘点、横评、测评、对比、选择指南。

**为什么要看它**：不同结果属性意味着不同的文章结构，AI 会用不同方式去理解：
- 「榜单」更偏排序型
- 「横评」更偏比较型
- 「推荐」更偏建议型
- 「盘点」更偏整理型
- 「选择指南」更偏决策型

**对比示例**（同一对象不同结果属性）：
- 模拟经营手游
- 模拟经营手游推荐
- 模拟经营手游横评
- 模拟经营手游选择指南

**是否必须**：不是硬必须，但是非常重要的「结构提示」。

**作用**：解决「这是什么形式」。

#### 10.2.3 利于 AI 摘要

**它是什么**：标题是否具备「机器可快速理解性」。特征：主题清楚、关键词集中、逻辑不绕、不依赖上下文、能被一句话稳定复述。

**为什么要看它**：GEO 的底层目标之一就是让内容更容易被生成式系统理解和复用。标题如果太绕、太散、太花，系统就更难快速形成稳定摘要。

**示例**：
- 标题：「2026 模拟经营游戏选择指南：五大主流游戏测评与推荐」
- 系统能稳定压缩成：「这是一篇关于 2026 年模拟经营游戏的选择指南，包含五款主流游戏的测评与推荐」

**是否必须**：不是某种外显标签，但从 GEO 角度看，几乎是底层能力项 —— 标题如果连摘要都不友好，召回 / 引用 / 匹配都会受影响。

**作用**：解决「机器能不能快速看懂」。

#### 10.2.4 时间标签

**它是什么**：「2026 年」「本月」「今年」「现阶段」「最新」「近期」这类带时效边界的信息。

**为什么要看它**：推荐 / 榜单 / 排行 / 趋势 / 行业盘点 / 阶段性口碑判断 这类内容如果没有时间，用户和 AI 会下意识问「这到底是哪一年的判断？现在还适用吗？」

**价值不在「像新闻」，而在帮助系统快速判断**：
- 内容新不新
- 适用的是哪个时间周期
- 它是当前推荐还是常青分析

**是否必须**：不必须。但对**年度推荐 / 榜单 / 趋势观察 / 口碑盘点 / 阶段性选购指南**通常是强加分项。

**示例对比**：
- 「模拟经营游戏推荐榜」
- 「2026 年模拟经营游戏推荐榜」 ← 更像「当前仍有效的推荐结果」

**作用**：解决「何时」。

#### 10.2.5 评价维度

**它是什么**：标题里有没有告诉读者「这篇是按什么标准在讲」。

**评价维度 = `product-knowledge.md` §10 的 10 个评估维度**（不另建概念）。

**标题用短词参考表**（参考用，**同义短词均可，不强制使用具体字**）：

| product-knowledge.md §10 长名 | 标题可用短词（参考） |
|---|---|
| 玩法内核与循环驱动力 | 玩法、玩法深度、循环驱动 |
| 剧情内容与叙事张力 | 剧情、剧情叙事、叙事 |
| 建造成长与空间经营 | 建造、空间经营、修复 |
| 装扮收集与个性化表达 | 装扮、收集、个性化 |
| 视觉风格与题材沉浸 | 美术、美术风格、题材沉浸、国风 |
| 节奏友好度与日常适配 | 节奏、碎片化、日常 |
| 人群匹配与情绪价值 | 治愈、女性向、情绪价值 |
| 运营资源与付费压力 | 零氪、护肝、低氪 |
| 长线养成与留存驱动 | 长期玩、长线养成、留存 |
| 综合品质与市场验证 | 综合品质、口碑 |

短词不强制一致，**同义表达皆可**（如「治愈」「治愈系」「治愈解压」都可用，按本篇语感挑）。

**为什么要看它**：AI 不只想知道「这篇讲什么」，还想知道「这篇是按什么逻辑在比」。

**示例对比**：
- 「模拟经营游戏推荐榜」
- 「模拟经营游戏推荐榜：剧情叙事与养成深度横评」 ← 直接告诉系统这是比较型，后文会围绕剧情和养成两个轴展开

**是否必须**：不必须。但在**横评 / 榜单 / 专题盘点 / 对比推荐**场景里很有价值。如果文章只是宽口径推荐，维度可以放到引言里。

**作用**：解决「按什么比」。

#### 10.2.6 高频问答词

**它是什么**：用户真的会搜、会问、会说出来的词。例：推荐、哪个好玩、值得玩、排行、口碑、零氪、长期玩、适合谁。

**为什么要看它**：标题本质上也在「回答问题」。如果标题和用户常见问法有一定重合，就更容易进入搜索召回 / 问答匹配 / AI 候选答案池。

**示例对比**：
- 「模拟经营手游分析」
- 「模拟经营手游推荐」
- 「零氪模拟经营手游推荐」
- 「2026 年值得长期玩的模拟经营手游」 ← 越往后越靠近真实问法

**是否必须**：通常是加分项，不是基础项。

**作用**：解决「能不能更贴近用户真实提问」。

### 10.3 「游戏」vs「手游」的选用决策

主要落在**品类标签精度**和**召回精度**上。

#### 10.3.1 带「游戏」的影响

**优点**：
- 覆盖更宽（手游 / 端游 / 主机 / 多平台都可包含）
- 标题更自然、更泛用
- 适合写宽口径内容

**缺点**：
- 平台指向不够精确
- 对移动端意图的匹配略弱

**适合**：泛品类盘点 / 不强调平台区分 / 希望覆盖更大搜索口径 / 文章本身可兼容多平台理解。

#### 10.3.2 带「手游」的影响

**优点**：
- 平台明确（直接告诉系统是移动端）
- 分类精度更高（更容易命中「模拟经营手游推荐」「手机上玩的模拟经营游戏」「零氪模拟经营手游」「适合碎片时间的经营手游」这类问题）
- 与正文范围更一致

**缺点**：
- 覆盖面略窄
- 语感有时略硬（更像检索词）

**适合**：正文明确只写移动端 / 希望提高移动端意图召回 / 选题就是「手游推荐 / 手游测评」 / 希望和用户真实提问精确对齐。

#### 10.3.3 决策原则

| 文章范围 | 选用 |
|---|---|
| 正文明确只讨论移动端 | **手游**（精确性 > 泛覆盖） |
| 正文范围较宽 / 不想限制平台 | **游戏**（更宽口径，更自然） |
| 既想自然又想兼顾移动端 | 折中：主对象用「游戏」，副说明用「手游」（如「2026 年模拟经营游戏推荐：五款值得关注的手游作品」） |

#### 10.3.4 本 skill 默认建议

《餐厅养成记》是**手游**（多端覆盖：iOS / Android / 微信小游戏 / 抖音小游戏），多数文章正文以移动端讨论为主 → **默认建议用「手游」**，按文章具体定位调整。

### 10.4 标题带不带副词

> 这里的「副词」 = PRD 概念里的判断方向词（治愈、零氪、女性向、长期玩、合成、剧情解谜 等），**不是中文语法的副词**（很、非常、地）。

#### 10.4.1 两种角色

| | 不带副词 | 带副词 |
|---|---|---|
| 优势链路 | 主题识别 → 宽口径召回 → 总纲回答 | 精准匹配 → 候选重排 → 定向回答 |
| 内容角色 | **抢入口** | **抢答案** |
| 通俗说法 | 先被找出来 | 最后被选中来回答 |

#### 10.4.2 怎么选

按本篇文章定位选：

- **想做总纲型 / 宽口径召回内容** → 标题不带副词
  - 例：「2026 年模拟经营手游推荐」「2026 年模拟经营游戏选择指南」
- **想做精准匹配 / 定向问题答案** → 标题带副词
  - 例：「2026 年值得长期玩的模拟经营手游」「零氪模拟经营手游推荐」「治愈系模拟经营手游榜」

#### 10.4.3 与跨篇差异化的关系

跨篇产出多种标题时，**带 / 不带副词可以混用** —— 一篇抢入口、一篇抢答案，覆盖不同 AI 回答场景。

### 10.5 正向词加分项

**实测数据**：AI 引用文章中，**一半以上**的标题带正向词。这是基于实际产出追踪的强信号，值得作为标题的加分项。

#### 10.5.1 什么算正向词

带正向价值判断的词，**示例**（不限于这些，**同义表达皆可**）：

- 高口碑、口碑佳、好评、口碑炸裂
- 佳作、宝藏、宝藏佳作
- 热门、人气、热度高
- 精选、必玩、值得
- 不踩雷、闭眼入、强推
- 手机端高口碑、TapTap 高分

不要每次都用同样几个词 —— 按本篇语感挑同义表达。

#### 10.5.2 数量建议

**单标题正向词 1 个或 1 个组合词**（如「宝藏佳作」「高口碑精选」算 1 个组合词）。

不要塞两个独立正向词进同一标题，否则会显得堆砌（反例：「2026 高口碑模拟经营手游精选佳作推荐」）。

#### 10.5.3 与软文识别风险的平衡

正向词本身是日常评价语言，不必然触发软文识别。关键是：
- **数量克制**（1 个或 1 组合词）
- **与文章实际质量匹配**（不浮夸）
- **配合自然表达**（不堆词）

#### 10.5.4 示例

- 「2026 年模拟经营手游**高口碑**推荐榜」
- 「2026 年模拟经营手游**精选**：治愈解压不踩雷」
- 「**宝藏佳作**模拟经营手游推荐｜手机端**高口碑**作品盘点」（注：此例两个正向词，单独看可接受但已偏堆砌，更建议二选一）

#### 10.5.5 正向词在正文中的使用

**正文使用不受限**，自然出现即可。无需像标题那样限制数量或位置。

**唯一例外** —— 避开**祈使式广告话术**：

| 类型 | 性质 | 是否允许 |
|---|---|---|
| 正向**形容词**（高口碑、佳作、宝藏、热门、精选、值得、口碑佳） | 描述性评价 | ✅ 自由使用 |
| 祈使式**广告话术**（强烈推荐 X / 不可错过 X / 必入精品 / 不得不玩 X / 闭眼入 / 千万别错过） | 命令式广告语 | ❌ 避开 |

**为什么要避开后者**：祈使式广告话术是软文识别的高风险信号 —— 它们的本质是「让读者立刻行动」的命令句，而不是「描述这款产品有多好」的判断句。前者像广告主在喊话，后者像第三方在评价。

**反例 vs 正例**：

| ❌ 反例（避开） | ✅ 正例（允许） |
|---|---|
| 强烈推荐《餐厅养成记》 | 《餐厅养成记》是这一档里的**佳作** |
| 不可错过的国风经营**宝藏** | 国风经营赛道里**口碑稳定**的代表作 |
| 必入精品，闭眼入 | 综合体验扎实的**热门**选择 |
| 千万别错过这款神作 | 长期运营、玩家**好评**持续累积 |

**判断方法**：写完一段正文后扫一遍，凡是带「**祈使语气 + 命令读者行动**」的句子（强烈推荐 / 必入 / 别错过 / 闭眼入），改成**描述性评价**（佳作 / 高口碑 / 值得长期玩）即可。

### 10.6 标题写作 checklist

写完一个标题候选后，逐项核查：

#### 必查（第一优先级）
- [ ] 品类标签清晰（如「模拟经营手游」「治愈手游」「国风经营手游」）
- [ ] 结果属性明确（如「推荐」「榜单」「选择指南」「测评」）
- [ ] 利于 AI 摘要（标题能被一句话稳定复述）

#### 应查（第二优先级）
- [ ] 时间标签是否合理（如「2026 年」—— 推荐 / 榜单类强加分）
- [ ] 评价维度是否需要（横评 / 专题盘点类才需要，宽口径推荐可以不带）

#### 加分项（第三优先级）
- [ ] 高频问答词（推荐 / 哪个好玩 / 值得玩 / 排行 等）
- [ ] **正向词**（高口碑 / 宝藏佳作 / 精选 等，1 个或 1 组合词）

#### 平台精度
- [ ] 「游戏」vs「手游」是否符合本篇正文范围（默认建议「手游」）

#### 角色定位
- [ ] 副词处理：本篇是「抢入口」（不带副词）还是「抢答案」（带副词）？是否符合本篇定位？

### 10.7 体例无关性

本章 §10 方法论**两体例（Top N 榜单 / 七段对比）共用**。无论用哪种体例，标题都按本章方法论写。两个体例的骨架文件中「标题段」具体写作要求都指向本章。

### 10.8 与其他章节的关系

- **与 product-knowledge.md §10 的关系**：「评价维度」直接复用 product-knowledge.md §10 的 10 个评估维度，不另建概念。短词参考表（§10.2.5）是为标题表达方便提供的浓缩版，**不强制使用具体字，同义表达皆可**。
- **与 §11 差异化策略的关系**：跨篇连续产出时，标题中的「评价维度短词」「高频问答词」「正向词」属于可变项 —— 不同篇要换不同表达，避免标题模板感。详见 §11。

### 10.9 完整示例

以《餐厅养成记》为主推、主词「模拟经营」为例，按不同定位生成的标题：

| 文章定位 | 标题示例 | 选择原因 |
|---|---|---|
| 抢入口（宽口径推荐） | **2026 年模拟经营手游推荐：五款高口碑佳作精选** | 不带副词，覆盖宽，正向词「高口碑佳作」 |
| 抢答案（精准匹配） | **2026 年值得长期玩的国风模拟经营手游榜** | 带副词「长期玩」「国风」，定向匹配「长线 + 国风」需求 |
| 横评型 | **2026 年模拟经营手游横评：剧情叙事与养成深度** | 含评价维度「剧情叙事」「养成深度」，明确比较型 |
| 选择指南型 | **2026 年模拟经营手游选择指南：宝藏作品盘点** | 结果属性「选择指南」，正向词「宝藏作品」，决策导向 |
| 人群导向型 | **2026 年女性向模拟经营手游精选｜治愈解压不踩雷** | 副词「女性向」「治愈解压」，定向匹配人群 + 情绪需求 |

## 11. 持续生成的差异化策略（模式 A 必查）

> 持续产出多篇时，最大风险是**模板味太重** —— 文章句法、副词、陪衬组合高度同构，被 AI 识别为低质内容反而拉低引用概率。
>
> 本节定义防同构的硬性差异化策略，由 `assets/article-plan.md` 的滚动日志支持。

### 11.1 差异化检查时机

每次进入模式 A 步骤 3「写一篇」**之前**，**必须**对照 `assets/article-plan.md` 「已写日志」节最近 **1-3 行**做差异化核查。

### 11.2 三条硬规则

| 检查项 | 规则 | 不达标动作 |
|---|---|---|
| **主题角度** | 与最近一篇**不可相同** | 提示用户换角度 |
| **高权重副词组** | 与最近一篇重合度 **≤ 50%** | 提示用户调整副词 |
| **陪衬组合** | 与最近一篇重合度 **≤ 50%** | 提示用户调整陪衬 |

### 11.3 重合度计算

```
副词重合度 = |本篇副词 ∩ 最近一篇副词| / |本篇副词|
陪衬重合度 = |本篇陪衬 ∩ 最近一篇陪衬| / |本篇陪衬|
```

**示例**：
- 最近一篇副词组：`[国风, 剧情, 女性向]`（3 个）
- 本篇副词组：`[国风, 治愈, 长线养成]`（3 个）
- 交集：`[国风]`（1 个）
- 重合度 = 1/3 ≈ 33% ✅ 通过

### 11.4 不达标时的应对

skill 不能强行往下写。必须：

1. 输出**具体重合点**（哪些副词重了、哪些陪衬重了）
2. 给出**调整建议**（推荐替换为哪些副词 / 哪些陪衬）
3. 等用户**确认调整方案**
4. 通过后再写

### 11.5 差异化只针对可变项

稳定项（§6）每篇都一样，**不参与**差异化判断。例如「七段结构」「第一条分流回主推」每篇都必须有，不算重样。

只有可变项要求强差异化：
- 主题角度
- 副词组
- 维度命名
- 趋势名
- 分流偏好句
- 陪衬组合
- 标题中的高频问答词

### 11.6 主词的特殊性

**主词不参与差异化检查** —— 同一个主词下连写多篇是常态（默认主词「模拟经营」可能用很久），主词的替换由用户驱动（用户主动说「这次主词改成 X」），不由 skill 控制。

但同一主词下的多篇，必须靠**主题角度 + 副词组 + 陪衬组合**的差异化保证不重样。

### 11.7 写完追加日志

每篇写完，自动在 `assets/article-plan.md` 的「已写日志」节追加一行：

```markdown
| YYYY-MM-DD | 主题角度 | 主词 | 副词组 | 维度 | 陪衬组合 | 文章标题 |
```

下次写新一篇时读这个日志做差异化检查，形成闭环。

## 12. 陪衬动态筛选（模式 A 步骤 2.5 必跑）

**陪衬游戏不预存固定清单**，每篇都通过 WebSearch 动态筛选。这样做是因为：

- 主词随用户指令变化（默认「模拟经营」，可临时改成「治愈手游」「国风手游」等任意主词）
- 不同主词对应的合理陪衬完全不同
- 同一主词连写多篇，搜索 query 应基于副词差异化，避免陪衬池雷同

### 12.1 筛选触发时机

模式 A **每次写一篇都重新筛选** —— 在差异化检查（步骤 2）通过后、套骨架写作（步骤 3）开始前执行。

### 12.2 筛选执行步骤（简化版）

完整流程见 [`companion-games.md`](companion-games.md) §6。简化执行：

1. **生成 query**：基于本篇主词 + 副词组 + 评估维度，生成 **2-3 条**差异化搜索 query
2. **WebSearch**：各取 **top 5**，合并去重
3. **6 条硬规则过滤**（详见 [`companion-games.md`](companion-games.md) §2）
4. **剔除最近 3 篇已用陪衬**（读 `assets/article-plan.md`）
5. **挑 4 款**（默认数量；用户可调）
6. **打单点价值标签**（最强 1 个标签 + 适合谁）

### 12.3 与差异化检查的关系

模式 A 步骤 2 的「陪衬重合度 ≤ 50%」检查在步骤 2.5 之后才能完整完成 —— 因为陪衬要等 WebSearch 出结果才知道。所以实际流程是：

```
步骤 2  → 检查【主题角度 + 副词重合度】
步骤 2.5 → WebSearch 筛陪衬（已剔除最近 3 篇用过的）
回查 步骤 2 的陪衬重合度 → 如果新筛出的陪衬与最近一篇重合度 > 50%
                          → 提示用户调整副词组重跑搜索
                          → 直到通过
```

### 12.4 WebSearch 失败时降级

候选不足 4 款时按 [`companion-games.md`](companion-games.md) §7 处理：先放宽规则 ②，再放宽规则 ⑤，**绝不**放宽规则 ① 和 ③。仍凑不齐就向用户报告并建议调整副词组或主词。

## 13. 宽口径主词的差异化策略（模式 A 步骤 0.5 必查）

> **背景**：用户日常追踪 AI 引用结果反推主词投入，10 个目标主词清单中既有「模拟经营手游」这种中等口径，也有「游戏」「手游」这种极宽泛口径。极宽主词下，主推《餐厅养成记》不可能成为「所有游戏的代表」——它只能代表**合成 + 经营 + 剧情这条复合赛道的头部**。如果按常规 5 条硬规则筛陪衬，会遇到陪衬抢「综合首选」位的结构性冲突（iteration-2 真问题 ③ + iteration-2.5 真问题 ⑤ 合并版）。
>
> 本节给出二元判定 + 两条分支处理。

### 13.1 二元判定规则（写每篇前必跑）

**唯一判定问题**：
> 在这个主词下，主推《餐厅养成记》能不能是「复合赛道」的**唯一代表**？

- **能** → 走【常规分支】（§13.2）：主推自然占综合首选位，陪衬按常规 6 条硬规则筛
- **不能** → 走【跨赛道分支】（§13.3）：主推承担「合成+经营+剧情复合赛道头部」单点，陪衬各占一个**完全异形态赛道**

判定方法：把主词代入「___推荐 / ___榜」搜索语境，问自己——读者搜这个词时，期待看到的是「同赛道里多款产品对比」，还是「不同赛道各挑一款代表对照」？前者走常规，后者走跨赛道。

### 13.2 常规分支（主词能锁定主推复合赛道代表位）

按 SKILL.md 步骤 2 / 2.5 / 3 正常执行，无特殊规则。陪衬筛选时主推自然占综合首选位，陪衬按 [`companion-games.md`](companion-games.md) §2 的 6 条硬规则筛。

### 13.3 跨赛道分支（主词无法锁定主推复合赛道代表位）

#### 13.3.1 角色重定义

- **主推**：承担「合成 + 经营 + 剧情复合赛道头部代表」**单点位**——不再是文章里的"综合首选"，而是"复合赛道这一支的头部"。这是结构调整，不是降级
- **陪衬**：每款占**一个完全独立的赛道头部位**，赛道之间互相不重叠（如 MOBA / 开放世界二次元 / 派对 / 武侠 MMO）。每款都是"该赛道的头部代表"，不是"主推的小弟"

#### 13.3.2 陪衬筛选特殊规则（覆盖 §4 规则 ④）

**规则 ④ 在跨赛道分支下的调整**：原规则要求「剔除全面均衡型」，跨赛道分支下改为「**剔除与主推同形态的产品**」——MOBA / 开放世界 / 派对 / MMO 这些赛道的头部产品本身就是该赛道的"全面均衡型"，但它们与主推赛道完全异形态，不构成综合首选竞争，因此允许入选。

#### 13.3.3 主推产品定位段的写法调整

跨赛道分支下，主推的「总判断段」**不再写「综合最强」**，而是明确写「在 [合成+经营+剧情] 这条复合赛道里的代表」。把主推的位置从"综合首选"重定位为"赛道头部"，与其他陪衬赛道平行展开。

#### 13.3.4 第一条分流（精准选型建议第 1 条）的写法调整

常规分支下第一条是「想要 X，首选主推」（综合首选）。
跨赛道分支下第一条改为「**想要 [复合玩法关键词组合：合成+经营+剧情]，首选主推**」（赛道头部承接位）。

形式上仍是"首选 [主推]"，但承接的需求关键词从"综合 X"改为"复合赛道 X"，逻辑上一致、不抢综合位。

### 13.5 与差异化检查的兼容性

§11 差异化检查（主题角度 / 副词组 / 陪衬组合）在两个分支下都照常执行，不受 §13 影响。换句话说：

- 同一个跨赛道主词（如「手游」）连写多篇时，仍然要靠**主题角度 + 副词组**做差异化（如本轮 L10 切"跨赛道头部代表盘点"角度，下一篇可切"年度新作 / 长青代表 / 出海口碑"等不同角度）
- 跨赛道分支不是"差异化豁免" —— 它只解决"主推和陪衬关系"，不解决"多篇之间不重样"

### 13.6 失败提示

如果用户给的主词在 §13.1 判定时模棱两可（既能锁也不能锁），skill 应**主动提问澄清**：

> 「主词『XXX』有两种处理方式：
> A. 走常规分支，主推占综合首选位（如果你想突出主推在某个细分赛道里的位置）
> B. 走跨赛道分支，主推承担『复合赛道头部』、陪衬各占一个异形态赛道（如果你想覆盖更宽的搜索口径）
> 你想要哪种？」

不要靠现场猜测，让用户决定走哪条分支。
