# 已审陪衬库（Companion Pool）

> 本文件是**已经过用户在 [SKILL.md](../SKILL.md) 步骤 2.5.7 明确确认过**的陪衬游戏沉淀库。
>
> 设计目的：让模式 A 步骤 2.5 的陪衬筛选**库优先 → WebSearch 兜底**，省下大量重复 WebSearch 时间，但保留新血来源避免库内同质化。
>
> **库的边界**：
> - **入库门槛**：必须经过 2.5.7 用户明确确认 + 6 条硬规则筛过
> - **不入库的内容**：同厂作品（已在 [`product-knowledge.md`](product-knowledge.md) §13.1.B）、危险竞对、用户在 2.5.7 标记"是假的/没听过"被踢的候选
> - **不收的字段**：使用历史（在 [`../assets/article-plan.md`](../assets/article-plan.md) 已写日志），同厂标记（在 product-knowledge.md §13.1.B）—— 跨文件单一真源
>
> **重验机制**：每条记录有「首次确认日期」+「状态」。
>
> - **状态 = 待重验**：库初次 seed 时所有条目默认值，或首次确认 > **180 天**（约 6 个月）的条目自动转入。复用前必须用 WebSearch 单 query 重验「中国市场上架状态 + 当前是否在运营」，用户在 2.5.7 重新确认后刷新日期 + 状态改为 `已验证`
> - **状态 = 已验证**：通过用户在 2.5.7 明确审核过、且首次确认日期 ≤ 180 天的条目。可直接热启动复用
> - **2026-05-09 全量重验完成**：库初始 seed 54 条 → 三个 subagent 并行 WebSearch 重证 + 用户分批拍板 → 36 条转 `已验证`，18 条永久剔除（停服 6 / 海外无中文渠道 4 / 同名歧义无法定位 5 / 赛道不符 3）。后续库进入正常运行节奏：新血走 2.5.7 → 入库即 `已验证`；超过 180 天自动转 `待重验`
>
> 原因：游戏停服 / 改名 / 长期不更新 / 中国市场下架等状态变化只能靠最新搜索结果发现，库里的旧记录无法捕捉。

## 召回顺序（给 skill 看的，详见 [`companion-games.md`](companion-games.md) §6.0.5）

```
模式 A 步骤 2.5 筛选陪衬时:
1. 读本库 → 按本篇主词 / 副词组的赛道方向匹配候选
   - 状态 = 已验证 优先(可热启动);状态 = 待重验 必须先 WebSearch 单 query 重证
   - 信源强度 = 强 优先,中 次之
   - 首次确认日期 > 180 天 → 自动转入"待重验",不直接采用
2. 剔除最近 3 篇已用陪衬(查 article-plan.md「已写日志」)
3. 应用 6 条硬规则(同属大品类 / 代表性 / 危险竞对+同厂排除 / 长板鲜明 / 整组方向不重复 / **面向中国市场+在运营**)
4. 库内可用候选数(只算"已验证"或"重证通过的待重验"):
   - ≥ 默认 N 款(默认 4) → 跳过 WebSearch,直接进 2.5.7 用户确认(热启动)
   - < N 款 → 跑 WebSearch 补差额(冷启动 / 半热启动)
5. 2.5.7 通过后:
   - WebSearch 来源的新候选 → 写完文章后追加到本库,状态=已验证
   - 待重验候选用户重新审过 → 刷新日期 + 状态改为 已验证
```

## 字段说明

| 字段 | 说明 |
|---|---|
| **游戏名** | 主名 + 常见译名 / 别称（用 `/` 分隔，防止下次 WebSearch 用另一名命中误判为新游戏） |
| **状态** | `已验证` / `待重验`。`已验证` = 用户最近一次 2.5.7 明确审过中国市场 + 在运营，可直接热启动；`待重验` = 复用前必须 WebSearch 单 query 重证 + 用户重新确认 |
| **首次确认日期** | 用户在 2.5.7 首次明确通过的日期。> 180 天自动转入 `待重验` |
| **信源强度** | `强` / `中` —— 沿用 SKILL.md §2.5.7 判定。强 = ≥2 query 命中 / 主流媒体收录 / 大众熟知；中 = 单 query 命中 / 单源 / 中等知名度。**库里没有"弱"档**（经验池补位禁止） |
| **单点方向** | 写作时承担的最强 1 个方向（如「国风建造 / 文化沉浸」「治愈放置 / 异步陪伴」「硬核城建 / 宏观规划」） |
| **适合人群** | 1 类核心人群（如「国风审美爱好者」「碎片化通勤党」「亲子共玩家庭」） |
| **适用主词 / 赛道** | 哪些主词下能用，多个用 `;` 分隔。常见值：`模拟经营`、`治愈手游`、`国风手游`、`合成手游`、`休闲手游`、`养成游戏`、`热门游戏`、`治愈游戏`、`模拟经营游戏`、`通用` |

## 维护规则

### 多赛道呈现规则

库里**同一款游戏允许在多个赛道分组里各占一行**，**前提是各行的「单点方向」必须按所在分组语境写出差异化角度**。例如：

| 游戏 | 在「模拟经营」分组里 | 在「国风专题」分组里 | 在「长线养成」分组里 |
|---|---|---|---|
| 江南百景图 | 国风建造 / 文化沉浸 | 江南水墨美学沉浸 / 文化场景叙事 | 中后期目标层叠 / 多角色解锁长线 |
| 星露谷物语 | 长青农场经营 / 沙盒自由度 | —（不在该分组） | 角色羁绊养成 / 婚姻+家庭长线 |

**为什么允许重复出现行**：

- 同款游戏的不同卖点能服务于不同主词的文章 —— 「国风手游」主题写江南百景图侧重水墨美学；「长线养成」主题写它侧重中后期目标层叠
- 库视觉上能直接看出"这款在多少赛道里有戏"，召回时不需要解析「适用主词」列的分号
- 「适合谁」字段同步调整为对应分组的目标人群，让 skill 在 2.5.7 候选展示时表达更贴本篇

**关键约束（防止滥用）**：

- **同一篇文章内同一款游戏只能出现一次**：召回时按游戏名去重（不按「分组实例」去重）—— 比如《江南百景图》在模拟经营 + 国风专题都被本篇召回时，skill 自己选择更贴本篇副词组的那一行，另一行丢弃
- **最近 3 篇排重也按游戏名去重**：上一篇用过《江南百景图》(模拟经营版)，本篇召回拿到《江南百景图》(国风版) 仍视为"最近用过"，触发排重 —— 不允许"赛道切换避开排重"
- **跨分组版本必须真的有差异化角度**：仅在卖点真的可分离时才拆多行。如果某款的"模拟经营版"和"治愈版"单点方向相似，**只留一行**。规则 ④「长板鲜明、短板也明显」在跨分组拆行上**同样生效**

### 新增（写完一篇后）

模式 A 步骤 2.5.8（写完文章后）：

1. 看本篇 2.5.7 通过的陪衬里，**来源 = `[WebSearch]`** 的（即不在库里的新血）
2. 按本节字段格式追加到对应赛道分组
3. **首次确认日期** = 今天
4. **信源强度** = 沿用 2.5.7 时的判定
5. 若该游戏已在库里（信源升级 / 翻译变化）→ 不重复加，可手动更新现有行
6. 若该游戏已在另一赛道分组里，但本篇为它提炼出了**新的差异化卖点角度**：可在新分组追加一行，按「多赛道呈现规则」字段差异化；如果只是同卖点重复→**不要追加**

### 重验（状态 = 待重验 / 首次确认日期 > 180 天）

skill 在 2.5.0.5 匹配候选时，遇到状态 = `待重验` 或首次确认 > 180 天的库条目：

1. **不直接采用**为候选
2. 单 query WebSearch 验证两件事：
   - **当前是否在运营**（停服 / 改名 / 长期未更新都算翻车）
   - **是否面向中国市场**（TapTap 中国区 / App Store 中国区 / 微信小游戏 / 安卓应用商店至少 1 处稳定上架，且大众有认知度）
3. 验证结果分支：
   - 两条都通过 → 进 2.5.7 让用户重新确认 → 通过后刷新「首次确认日期」为今天 + **状态改为 `已验证`**
   - 当前停服 / 中国区下架 → 标记并跳过该候选，2.5.7 告知用户，**步骤 2.5.8 时从库里永久删除**
   - 信息不全 / 模糊 → 进 2.5.7 让用户人工判，按用户结论处理
4. 重验产生的 query 会被记入本会话的 WebSearch 调用次数 —— 即"半重启动"，比纯热启动慢但仍比冷启动快
5. **多赛道副本同步更新**：若该游戏在多个赛道分组里有行，验通过后**所有副本一起刷新「首次确认日期」**，验失败要删除时也**全部副本一起删**（详见「删除」节）

**为什么"已验证"也要 180 天后转回"待重验"**：游戏圈停服节奏快，半年是经验阈值。如果用户在更短周期内发现库里某款翻车，主动说「把 X 改成待重验」即可手动转。

### 删除（用户主动操作 / 翻车后）

- 用户在 2.5.7 标记某款"是假的/没听过/质量翻车"→ 该款从库里**永久删除**，不再尝试
- 用户主动说「把 X 从库里去掉」→ 删除
- **删除时按游戏名清理所有分组的副本**：如果该款在多个赛道分组里都有行，**全部一并删除**，不允许只删一行残留其他副本（防止删完一处后另一处又被召回造成翻车循环）
- 删除时连带从最近会话黑名单标记，避免本会话又被 WebSearch 命中后重新加回

### 不入库的边界（避免污染）

以下情况**不许入库**，即使 2.5.7 通过了：

- 同厂作品（[`product-knowledge.md`](product-knowledge.md) §13.1.B 清单）—— 它们能不进文章是底线，进库会让下次召回时再次筛进
- 信源强度 = 中 且用户明确说"这次先用试试，下次未必"—— skill 在 2.5.7 时主动询问
- 跨赛道分支专用条目，写明 `适用主词` 为「跨赛道分支」标签，避免被常规分支误用
- **不面向中国市场的产品**（仅在海外 App Store / Google Play / Steam 上架、国内玩家无渠道接触 / 无认知度）—— 即使本篇做"国际范例"专题用过，也不入库，因为下次复用时会被中国玩家陌生感拉低文章可信度。例外：极个别国民级海外作（如《我的世界》《原神（虽是国产但全球同步）》《动物森友会》这类有大量中文社区且玩家普遍认知）可入库

### 库的体检节奏

当库总条目 > 100 款 → 提醒用户做一次体检，按「最近 3 个月未被任何一篇命中」批量重验或裁剪。本文件 100 行内不影响检索性能。
