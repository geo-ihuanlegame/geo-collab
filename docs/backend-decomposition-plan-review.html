★给管理层的一句话结论

大方向是对的，可以做；但原方案的「时间表」和「怎么切」两处过于乐观，照原样推进会踩坑。

把单体后端拆开、让生文/发文团队各管一摊、能独立上线 —— 这个目标本身值得投入。 但我们逐行核对代码后发现：① 原方案说的「代码基本不用搬家、解耦只要 4–6 周」，实测要 2–3 倍时间； ② 原方案把业务切成的「5 个独立域」里，有两组在代码里是焊死在一起的，硬拆会把简单问题变成分布式难题。 建议采纳方案的方向，但把目标从「一口气拆 5 个」改成「先稳妥拆 2 个大块」，工期与风险都更可控。

方案边界与代码现实冲突

1先用大白话讲清：这方案到底在干什么

现在整个后端是「一栋大房子里所有人挤在一个大厅办公」——17 个功能模块跑在同一个程序、共用一个数据库。 好处是简单，坏处是：改其中任何一处（哪怕只改生文），都要把整栋楼重新装修上线；人多了（4–6 人）在同一片代码上互相踩脚。

原方案想做的，是把大厅隔成 5 个独立办公室（内容、生成、发布、账号、平台），每间屋自己装修、自己开门、自己换人，互不打扰； 门口再设一个统一前台（API 网关）负责验证身份和分流。前端用户那一侧看起来完全不变，还是一个后台。

这个思路在行业里是成熟做法，方向我们认可。问题出在两个具体判断上，下面用代码证据说明。

2方案的优点（这些判断是对的）

✓ 站得住的优点

  • 解决的痛点是真实的：改一处要全量重发、多人抢同一片代码冲突、一处出问题拖垮全站 —— 代码现状确实如此。
  • 「先逻辑解耦、再物理拆分」的稳健路径正确：不一上来就拆进程，先在单体内把边界划清楚，每步可上线、可回滚 —— 这是降低风险的正确姿势。
  • 前端不拆，对的：前端用户无感、改造面最小，把复杂度都收在后端。
  • 技术栈策略克制、理性：Python 守住浏览器自动化/AI/富文本这些「深水区」，Go 只切网关、热榜这类边缘，不盲目全量重写。
  • CI 护栏思路好：加门禁规则禁止「新增跨域耦合」，避免边重构边制造新问题 —— 这条应当立即落地。

△ 但要付的代价（方案自己也承认）

  • 整体复杂度上升：从 1 个程序变成 5 个 + 网关 + 消息总线，运维、监控、排错成本都上一个台阶。
  • 分布式带来「最终一致」难题：原来一个数据库事务能保证的事，拆开后要靠对账、幂等、重试来兜底，调试链路变长。
  • 一次报错可能要翻 5 个服务的日志才能定位。
  • 本地开发、新人上手门槛升高。
  • 周期与人力投入实打实，且（见下）比方案估的更长。

3风险一：工期被低估了 2–3 倍

原方案的核心卖点是「代码已按模块分好，拆分时代码基本不用搬家」「P0 逻辑解耦只要 4–6 周」。 我们实测代码后，认为这个判断过于乐观。拆服务的难点从来不是「搬代码」，而是「拆开后那些原本在一个程序里悄悄完成的协作，全要改成跨网络调用」。下面是体检数据：

现状体检（实测数字）说明它为什么拖慢拆分
35+ 处跨模块直接调用一个模块直接 import 另一个模块的代码/数据，多达 35 处以上。拆开后每一处都要改成「跨服务网络调用 + 失败重试」。
核心「用户」数据被 7 个模块共用几乎每个模块都靠外键挂在「用户」表上。拆库时这是绕不开的硬骨头，要么共享、要么复制（带来一致性风险）。
52 条数据库变更脚本是单一条链所有模块的建表/改表历史串在一条线上。拆成多服务后，多人并行改库会频繁撞车，需要先拆这条链。
165 个测试文件假设「单进程一个程序」现有约 750 处测试都基于「所有模块在一个程序里」搭建。拆服务后这套测试要大面积重写，否则就是裸奔上线。
⏱ 工期重估：原方案说「P0 解耦 4–6 周、10–15 周达成核心目标」。 按上面真实耦合度，仅「逻辑解耦」一项就要 8–12 周（不是 4–6 周）；叠加测试重写（3–4 周）、生成模块的改造（见风险三）， 达成「发文/生文能独立部署」这个核心目标，更现实的预期是 4–6 个月，而非 2.5–3.5 个月。

⚠️ 这不是说方案不可行，而是建议给老板汇报和排期时用更稳的数字，避免承诺了 3 个月、实际半年，影响团队信誉与业务节奏。

4风险二：方案画的「5 个域」里，有两组在代码里是焊死的

这是最重要的一条。拆房子要沿着「墙缝」拆，不能从「承重墙」中间劈开。原方案按功能名字画了 5 个域， 但代码里有两组模块共用了「承重墙」（共享的数据外键、共享的磁盘文件、共享的状态流程），硬拆会把本来简单的事变成最难的「分布式事务」。

① 「账号」和「发布」被原方案拆成两个独立域 —— 但代码里它们物理不可分

发布记录直接用外键挂着账号（publish_records.account_id）；发文时还要去硬盘上读账号登录时写下的浏览器登录态文件， 两边共用同一块磁盘目录、共用一把跨进程的「浏览器锁」。这意味着把它们拆成两个独立服务，就要在两台机器间同步加密文件、跨网络抢锁 —— 徒增故障点。

结论：账号 + 发布应当合成「一个发文域」，而不是方案里的两个独立域。

② 「内容」和「生成」被原方案拆成两个独立域 —— 但它们共用同一套「审核/成组」流程

生文、智能体编排、自动评审这三路，最后都会调同一个函数把文章标成「待审」并归到同一个分组（mark_pending_and_group）， 还都要写文章的「审核状态」、同步文章正文的三份并行副本。这套流程现在靠「在同一个程序、同一个数据库事务里完成」来保证不出错。

一旦把「内容」和「生成」拆成两个服务，这套原子操作就被网络切断，要靠分布式事务/补偿来兜底 —— 这是性价比最低的拆法。 结论：内容 + 生成 + 编排短期应当留在「一个内容域」内，对外只暴露一个「成组/送审」的统一接口。

✅ 校正建议：把目标从「一口气拆 5 个域」改成「沿真实墙缝先拆 2 个大块」
原方案的 5 个域按代码现实校正后判定
内容 + 生成（A、B 两个域）合并为 内容域（文章/生文/编排/模板/图库/评审）原拆法会制造分布式事务
发布 + 账号（C、D 两个域）合并为 发文域（任务/驱动/账号/浏览器/worker）原拆法物理不可分
平台（E 域）保留 平台/身份（用户/鉴权/审计/热榜）这个切得对

先稳妥拆成 内容域 / 发文域 / 平台域这 2–3 个大块，就已经能满足「生文、发文团队各自独立上线」的核心诉求； 等某个域真的遇到扩容瓶颈，再沿清晰接口进一步细拆。不必、也不该一开始就拆 5 个。

📎 代码证据（供技术同事复核）：tasks/service.py:21 与 articles/service.py:41 双向引用、 pipelines/executor.py:16 调 mark_pending_and_group、accounts↔tasks 外键与共享浏览器 profile、 Asset 表被 articles/accounts/tasks 三模块外键引用。

5风险三：「生成」改独立服务，是一次重写，不是搬代码

方案把「生成」轻描淡写成「拆成一个独立服务」。但代码里，生文和编排根本没有独立的后台进程—— 它们是直接在 API 网页服务器内部「开线程」干活的（接到请求 → 当场在本进程里起线程跑生成 → 立即返回）。

要把它变成独立服务，等于要把这套「当场开线程」改成「请求 → 丢进任务队列 → 后台 worker 去捞」的全新架构。 这是地基层面的重写，不是「把文件夹挪个位置」。同时还有几处「只能单实例运行」的硬约束（启动时会把所有在跑的任务无条件判死复位）， 意味着不能简单地多开几个实例来扩容，得先补上分布式锁/租约机制。

📌 提示：这块隐藏工作量是原方案完全没计入的，也是上面「工期 ×2–3」的主要来源之一。 建议把它排到最后做、且只在真有扩容需求时才做——短期让它留在内容域里按现状跑，零改造、零风险。

7给管理层的决策建议

一句话：批准这个方向，但按下面三条校正后再启动，能省下大量返工。
