业务重构方案 · 决策终稿

把单体平台演进成「模块可独立部署、团队可独立维护、并向大模型开放能力」的架构。本稿在通读 2.3 万行后端代码、并逐项核对早期「5 域拆分」设想的基础上，给出务实的拆分路线、真实工期、面向大模型的 MCP + 技能库规划，以及需要拍板的决策。

★ 一句话结论

拆分这件事值得做、方向正确：现有代码确实支持「业务模块独立部署、团队各管一摊」。但有两条要先校正——

① 不要一步切成十几个微服务，那会用分布式复杂度拖垮一个还在快速迭代的产品；推荐先粗粒度拆 4~5 个可独立部署单元 + 模块化单体起步 + 按需再细分。

0 这份方案是怎么来的：我们核对了早期设想

为避免「拍脑袋」，本稿先对早期的《后端 5 域拆分》设想做了代码核对，再据实修正。

🔬 核对结论：早期「5 域」方向对，但两处边界与代码现实冲突

早期设想把后端切成「内容 / 生成 / 发布 / 账号 / 平台」5 个独立域。逐行核对代码后发现，其中两组在代码里是焊死在一起的——硬按 5 域拆，会把本来简单的事变成最难的「分布式事务」：

早期设想的边界代码里的真实情况校正
「账号」与「发布」拆成两个独立域 发布记录用外键直挂账号；发文时要读账号写在硬盘上的浏览器登录态文件、共用同一把跨进程锁 → 物理不可分 合并为发文域
「内容」与「生成」拆成两个独立域 生文 / 编排 / 评审三路共用同一套「审核+成组」原子流程，靠同库同事务保证不出错 → 拆开即制造分布式事务 合并为内容核心
「平台」（用户 / 鉴权 / 审计 / 热榜）独立 低耦合、天然独立 这个切得对

📎 代码证据（供技术复核）：tasks/service.py↔articles/service.py 双向引用；pipelines/executor.py 调 mark_pending_and_group；accounts↔tasks 外键 + 共享浏览器 profile；Asset 表被 articles/accounts/tasks 三模块外键引用。

1 现状盘点（一句话版）

我们实际拥有什么——这是所有决策的事实基础。

后端

FastAPI + MySQL 单体，12 个活跃业务模块、42 张表、约 2.3 万行业务代码。模块之间耦合紧密：内容（文章）和发文双向依赖，账号与发文不可分割，多处「跨模块同一笔事务」。

前端

React + 自研组件，目前没有 URL 路由——所有页面靠一个总开关切换，地址栏永远不变，无法直达/分享某个页面。未使用任何成熟组件库。

团队协作

已规范使用 Git 提交规范 + PR 流程 + CI 门禁，基础很好。但缺少「代码归属」机制，多人改同一片代码时容易冲突；数据库迁移是单一链路（52 条串成一条），并行开发常撞车。

部署

Docker Compose 一套镜像跑全部，web 与发文 worker 共用同一镜像。发文依赖浏览器自动化（Linux 专属），且必须单实例运行。

2 核心决策：粗粒度，而非「微服务全家桶」

这是本方案与「直接上 12 个 Go 微服务」最大的区别，也是最需要老板拍板的地方。

⚠️ 为什么不建议一步到位 12 微服务

十几个微服务会同时引入：消息总线、分布式锁、跨服务事务、十几条流水线、十几个镜像、还要用 Go 重写包括复杂富文本转换在内的所有逻辑。这套「分布式税」对 2~3 人团队是净负债，且每个跨服务调用都新增一种出错可能。投入大、风险高、见效慢。

✅ 推荐：粗粒度 + 按需抽离

沿代码真实的「断裂面」切成 4~5 个单元，而不是沿目录硬拆。把强耦合的模块留在同一单元里同库同事务（最省心），把天然独立的（发文、热榜、身份）先拆出去。每一步都能独立上线、交付价值、可回滚。需要更细时再继续抽。

3 目标架构：4~5 个可独立部署单元

按真实耦合划分（即上文校正后的边界）；每个单元有清晰的负责人与边界。

内容核心 保留 Python
文章 / AI 生文 / 工作流编排 / 提示词 / 图片库 / 审核。这些共享「文章状态机+富文本」，强行拆开就是制造分布式事务，故同库同事务。
发文域 保留 Python
发布任务 / 账号 / 浏览器驱动 / 发布 worker。绑定 Linux 浏览器自动化、加密凭据、单实例约束——天然独立，第一阶段即可独立部署。
基础身份 可 Go
用户 / 平台 / 心跳。全员依赖，先稳定契约。
热榜 已独立
第三方热榜聚合，现状已是独立 Node 子服务，零成本。
网关 / 聚合 可选
统一入口、鉴权、路由，作为渐进迁移的「接缝」。
Agent 接入层 已有 POC
MCP 网关（大模型调后端的桥）+ 技能库（给大模型装的 skill 包）。已有可用雏形，本阶段产品化，详见第 4 节。
保留 Python 复杂事务/浏览器绑定，短期不重写 可 Go 低耦合域，适合用 Go 练手并受益 可选 按需引入

Go 语言的定位

不是「全量重写」，而是按性能与团队边界按需抽离。身份、热榜、数据统计这类近乎纯增删查的低耦合域，是用 Go 练手并受益的首选；内容核心与发文域的复杂事务短期保留 Python 最稳。把「Go 微服务」当作可达的终态，而不是起步的硬要求。

4 面向大模型：MCP 网关 + 技能库（已有 POC，本阶段产品化）

新增需求：让大模型通过 MCP 调用后端服务，并提供一个技能库方便把整理好的 skill 给大模型安装。好消息是这两块仓库里都已有可用雏形——本节说明现状、在新架构里怎么摆、以及如何从 POC 走向生产。

✅ 关键判断：不是从零造，而是「产品化 + 纳入新架构」

现有 MCP 服务已封装 22 个工具（读 10 / 写 8 / 评估回流 4），且本来就是以「HTTP 调后端 API」的薄封装方式实现的——这意味着拆服务后它天然就是一个独立网关，几乎无需重写。技能库也已有一个带版本、可一键安装、带校验的 bundle。所以这块是中等投入的增强，不是另起炉灶。

① MCP 网关：大模型调用后端的「桥」

维度现状（已有）产品化要补
是什么 独立 MCP 服务（FastMCP），把后端能力包装成 22 个工具给大模型调用；同进程挂在 /mcp，也有命令行入口可独立跑。 在新架构里独立成一个无状态网关单元（归入「Agent 接入层」），可独立扩缩、独立发版。
怎么连后端 工具内部是用 HTTP「自调」后端接口实现的，不含业务逻辑。每个业务域还各有一组带 MCP 令牌保护的专用接口。 把「内部接口地址」从单体改成分别指向内容核心 / 发文 / 身份各服务即可——改动小，是这套架构的最大红利。
鉴权 用独立的服务令牌（GEO_MCP_TOKEN），与用户登录态完全隔离，恒定时间比对。 当前是全局共享一把令牌（POC 够用）；多团队/多租户要升级为按用户/按租户的令牌（可吊销、可审计）。
可观测 / 稳态 已有 MCP 专用错误映射（上游错→可重试码 / 其它→500），错误对大模型友好。 补全链路追踪（一次大模型调用串起多服务日志）、调用限流/配额、工具 schema 版本化与前向兼容。

② 技能库：给大模型安装的 skill 仓库

⚠️ 先澄清命名：这跟早已下线的后端 DB「skills 模块」完全是两回事。要做的是 面向大模型的 skill 包仓库（每个 skill = 一个目录：说明书 + 参考资料 + 骨架模板），由大模型按需安装到自己的工作目录。

维度现状（已有）产品化要补
是什么 已有一个带版本号的技能 bundle（当前版本含「写作 / 校验 / 编排」三个 skill + 一个入口命令），并附 sha256 校验、CI 防篡改。 从「一个 bundle 打包三件」升级为真正的注册表：可列出 / 搜索 / 按需安装单个 skill，各自独立版本号。
怎么安装 大模型调一个 MCP 工具即可一键把 skill 包写进自己的 .claude/ 目录；也支持打包 zip 下载。 补 列出 / 安装 / 更新 全套，带版本兼容性校验（skill ↔ 所需 MCP 工具最低版本）。
治理 仓库根另有若干参考 skill 设计；bundle 由服务端统一打包。 建入库审核流程 + stable / beta 双发布渠道；每个 skill 声明依赖哪些 MCP 工具，装时校验网关是否提供。
放哪 — 技能库 = 静态包 + 一个轻量注册表接口，和 MCP 网关同属「Agent 接入层」一起部署最自然。

💡 MCP 网关与技能库是一对：网关给「工具」，技能库给「用好这些工具的配方」。两者都属面向大模型的边缘层，无状态、与核心业务解耦，是这套架构里最容易独立演进的一块。

5 工期现实检验：为什么是 4~6 个月，而不是 3 个月

拆服务的难点从来不是「搬代码」，而是「拆开后那些原本在一个程序里悄悄完成的协作，全要改成跨网络调用」。下面是后端代码体检的实测数字。

体检项（实测）为什么它拖慢拆分
35+ 处跨模块直接调用一个模块直接引用另一个模块的代码/数据。拆开后每一处都要改成「跨服务网络调用 + 失败重试」。
核心「用户」数据被 7 个模块共用几乎每个模块都靠外键挂在「用户」表上。拆库时绕不开，要么共享、要么复制（带一致性风险）。
52 条数据库变更脚本串成一条链所有模块的建表历史串在一条线。多服务并行改库会频繁撞车，需先拆这条链。
165 个测试文件假设「单进程一个程序」约 750 处测试都基于「所有模块在一个程序里」搭建。拆服务后要大面积重写，否则就是裸奔上线。
「生成」没有独立后台进程生文/编排是直接在网页服务器里「当场开线程」干活的。改成独立服务等于换成「任务队列 + 后台 worker」的新架构——是地基重写，不是搬文件。

⏱ 工期重估：早期设想「解耦 4~6 周、10~15 周达成」。按真实耦合度，仅「逻辑解耦」一项就要 8~12 周；叠加测试重写（3~4 周）与生成模块改造，达成「发文/生文能独立部署」的核心目标，更现实的预期是 4~6 个月。这不是说不可行，而是建议对外用更稳的数字承诺，避免承诺 3 个月、实际半年，影响团队信誉与业务节奏。

6 三个诉求如何满足

逐一回应最初提出的三点目标。

① 业务模块相互独立部署，方便维护

发文域第一阶段就独立部署、独立发版、独立扩缩，发文团队完全自治。生文/编排先在「内容核心」内逻辑独立、留好抽离接口；当生文确实需要单独扩容时，再沿这一个接口抽成独立单元。先模块化、保留接缝、按需再抽

② 前端菜单页独立 URL 路由 + 是否引入 ant-design-pro

现状是「无路由单页」，地址栏永远不变。建议分两步：

步骤做什么工期 / 风险
第一步
推荐先做		引入路由库（react-router），保留现有全部组件。把每个菜单映射成唯一 URL（如 /content/pending），可直达、可分享、可加书签——直接满足诉求②。约 2~3 周
中等
第二步
按需评估		是否上 ant-design-pro。它能省掉大量列表/表单代码、自带后台皮肤与权限模型；但属重型框架，富文本编辑器、节点编辑器等自研组件仍需保留，且全量迁移工期长。约 5~8 周
较高

对 ant-design-pro 的建议：不要为了「换框架」而换。 如果主要诉求是「表单/列表更规整 + 现成后台模板」，可以只按需引入 antd 组件库，路由用轻量方案即可，无需迁到 UmiJS 全家桶。等第一步上线后用真实痛点再决定是否深入。

③ 多人协作如何用 Git 避免冲突

团队已有良好的提交规范与 PR 流程。避免冲突的真正杠杆有四个：

措施作用
保持单一代码仓库
（monorepo）		2~3 人用多仓库是净负担。一个仓库 + 按单元分目录，统一管理最省心。
补「代码归属」配置
（CODEOWNERS，当前缺）		核心手段：人按目录分工 → 改的是不同文件 → 天然不撞车。冲突只剩「公共契约 / 共享代码」，而那本就该多人评审。
主干开发 + 短命分支沿用现有 PR 流程，特性分支控制在 2~3 天内合并，避免长命大分支积累冲突。
拆分数据库迁移链当前单一迁移链是并行开发频繁撞车的隐藏根源。每个单元各管自己的迁移，互不干扰。

7 迁移路线（每步独立上线、可回滚）

不追求「一次切完」，追求「每步交付价值」。右侧为粗估工期。

  1. 1~2 周地基（不动业务）：补代码归属配置（CODEOWNERS）+ 拆迁移链；前端引入路由打通独立 URL。✓ 立即交付诉求②，零业务风险
  2. 6~10 周发文域独立：把发文相关模块整体打包成独立部署单元（保留 Python，不重写），仅改三处对接点。✓ 交付诉求①核心，发文团队自治
  3. 4~6 周内容核心收口：把文章状态机统一成单一写入口 + 对外原子接口；内容与发文之间通过契约/事件清晰解耦。✓ 两大单元彻底分清边界
  4. 3~5 周·可并行Agent 接入层产品化：MCP 网关独立成无状态单元、内部地址改指向各域服务；技能库从单 bundle 升级为注册表 + 按用户令牌。✓ 大模型可经 MCP 调各服务、可装 skill；与上面阶段可并行
  5. 按需按需 Go 化：身份、热榜、数据统计等低耦合域按团队意愿用 Go 抽离、练手。✓ 渐进引入 Go，风险可控
  6. 按需按需扩展：哪个单元真遇到性能瓶颈，才为它付「多实例 + 物理拆库」的成本；生成服务化、前端 ant-design-pro 均在此阶段按需评估。✓ 只在真有需要时投入

⏱ P0→P2 合计约 4~6 个月达成「发文/生文独立部署」核心目标；P3、P4 为增量优化，不阻塞上线。

9 需要老板拍板的决策

下面每一项给出推荐选项，确认后即可启动。

  1. A. 拆分粒度：粗粒度（4~5 单元）还是细粒度（10+ 微服务）？
    ✅ 推荐：粗粒度起步、按需再细分。
    备选：直接上细粒度微服务——投入与风险显著更高，不建议当前规模采用。
  2. B. 工期与承诺：按 4~6 个月稳健排期，先交付发文域独立？
    ✅ 推荐：是。以「发文域独立部署」为第一个可见里程碑，滚动推进。
    备选：压到 3 个月——与代码体检结论不符，存在跳票风险。
  3. C. Go 的引入节奏：终态可选 vs 起步即全量 Go？
    ✅ 推荐：低耦合域先用 Go 练手，核心域短期保留 Python，按需推进。
    备选：全量 Go 重写——需评估重写成本与人力储备。
  4. D. 前端：先打通独立 URL 路由，ant-design-pro 按需再评估？
    ✅ 推荐：第一步轻量路由化（2~3 周，直接满足诉求②）；是否上 Pro 上线后再定。
    备选：一步到位迁 ant-design-pro 全家桶（5~8 周，较高风险）。
  5. E. 是否立即启动两项「零风险地基」？
    ✅ 推荐：现在就做——① 补代码归属配置（CODEOWNERS）② 拆数据库迁移链。立竿见影减少多人冲突，无业务风险。
  6. F. 面向大模型的 MCP + 技能库：在现有 POC 上产品化？
    ✅ 推荐：是。MCP 升级为独立网关 + 按用户令牌，技能库升级为注册表；与发文域拆分并行推进，不占核心工期。
    备选：维持现状 POC（全局令牌 + 单 bundle）——内部小范围够用，但不适合多团队/对外。
