Geo 协作平台 — Go 微服务重构方案

把现有 Python 单体演进为「模块独立部署、团队各自维护、契约 + 异步事件协作」的微服务体系,统一管理后台用 ant-design-pro 重构。本文为决策层概览。

📅 2026-06-24 🎯 后端 Go 微服务 🔁 绞杀者渐进迁移 🗄 共享库软隔离起步 🐍 发文保留 Python

✓ 结论(先看这里)

现有系统是一个功能完整、依赖关系清晰(无环)的 Python 单体。具备拆分为微服务的条件,但不建议推倒重写。推荐采用「绞杀者」渐进迁移:保持业务在线,新 Go 服务逐个上线接管,旧单体逐步退役,全程可回滚、不中断业务。

  • 能达成核心诉求:生文、发文、编排等模块最终各自独立部署、由对应团队独立发版。
  • 关键取舍:发文/浏览器自动化(Playwright + 远程接管)保留 Python,封装成独立发布服务——重写它收益低、风险高。
  • 最大价值早期可见:统一管理后台 + 发文团队自治,可在迁移中段就交付,无需等到全部完成。
  • 预计周期约 10–13 个月(团队并行推进),按里程碑分阶段交付,每阶段独立产生价值。

1 三条决策基线

以下三个方向已确认,它们决定了整套方案的成本与风险走向。

发文技术栈
浏览器自动化保留 Python。 Playwright + Xvfb/noVNC 远程接管 + 各平台驱动是 Linux 重资产,封装成独立的「发布服务」,与 Go 服务通过契约/事件协作。重写风险最高、收益最低,不做。
数据库策略
先共享一个数据库、按表所有权软隔离,再逐步物理拆库。 每张表只有一个服务可写,禁止跨服务直连。迁移平滑、回滚容易;边界验证稳定后再拆独立库。
迁移节奏
绞杀者(Strangler)渐进迁移。 新增 API 网关作「接缝」,Go 服务一个个上线接管流量,Python 模块逐个退役。每步可独立验证、可回滚,业务不中断。

4 优势与代价

微服务化解决"团队协作 + 独立部署",但也引入分布式系统固有的复杂度。客观权衡如下。

✅ 收益

  • 模块独立部署、独立发版:生文、发文、编排互不阻塞,团队各自掌控发布节奏。
  • 团队边界清晰:每个服务有明确 owner 与契约,降低协作摩擦与代码冲突。
  • 按需扩缩:生文/编排可水平扩展(解除现有"只能单实例"限制),发文按平台独立扩容。
  • 故障隔离:单服务故障不再拖垮全站;一个模块崩溃不影响其它业务。
  • 技术栈灵活:Go 高并发低成本承载核心域,Python 留守其最擅长的浏览器自动化与 AI 生态。
  • 统一现代后台:ant-design-pro 提升运营体验与可维护性。

⚠️ 代价

  • 分布式复杂度上升:跨服务事务由本地事务变为 Saga/事件,引入最终一致与新的失败模式。
  • 运维成本增加:多服务部署、链路追踪、监控告警是必备投入,不能省。
  • 迁移周期较长:绞杀者模式稳但慢,需长期 Go/Python 共存。
  • 团队 Go 能力要求:需要一定的 Go 工程储备(可用 go-zero + 低风险服务先练手)。
  • 部分逻辑需重做:生文/编排的并发机制依赖单进程内存,跨服务后必须改造为分布式版本。

5 关键风险与应对

基于真实代码探查识别的主要风险,均有可落地的缓解措施。

高
富文本转换器移植文章正文存三份并行结构(编辑器/HTML/纯文本),转换逻辑复杂,Go 重写易引入内容错误。 → 应对:初期保留 Python 转换组件作为旁路,稳定后再评估是否移植。
高
并发机制重做生文/编排现依赖单进程内存锁与计数器(故只能跑单实例)。 → 应对:集中改造为 Redis 分布式锁 + 租约/心跳机制,改造后即可多实例水平扩展。
中
跨服务事务一致性建发布任务、分发、审核状态变更原本是单库事务。 → 应对:审核状态收归单一服务独占写入(消除多方竞争)+ Saga + 事件幂等;难点事务单独评审设计。
中
发文仍受单实例约束浏览器 profile 跨进程锁机制决定发布 worker 暂不能水平扩展。 → 应对:接受现状,采用主备高可用;长期可加租约后再放开。
中
迁移周期与团队磨合绞杀者周期长、Go 经验需积累。 → 应对:先用低风险支撑域服务练手跑通全流程;每阶段独立交付价值,允许长期共存。
