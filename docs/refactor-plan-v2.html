把单体平台演进成「模块可独立部署、团队可独立维护」的架构。本方案基于对现有代码的逐项核实，给出务实的拆分路线与需要拍板的决策。

★ 一句话结论

现有代码支持「业务模块独立部署」这个目标，但以当前 2~3 人团队的规模，不建议一步切成十几个微服务——那会用分布式系统的复杂度拖垮一个还在快速迭代的产品。

推荐：先做粗粒度拆分（4~5 个可独立部署单元）+ 模块化单体起步 + 按真实需求再细分。发文域第一阶段就能独立部署、独立发版；Go 微服务作为「终态可选项」按需推进，而非起步门槛。

1 现状盘点（一句话版）

我们实际拥有什么——这是所有决策的事实基础。

后端

FastAPI + MySQL 单体，12 个活跃业务模块、42 张表。模块之间耦合紧密：内容（文章）和发文是双向依赖，账号与发文不可分割，多处「跨模块同一笔事务」。

前端

React + 自研组件，目前没有 URL 路由——所有页面靠一个总开关切换，地址栏永远不变，无法直达/分享某个页面。未使用任何成熟组件库。

团队协作

已规范使用 Git 提交规范 + PR 流程 + CI 门禁，基础很好。但缺少「代码归属」机制，多人改同一片代码时容易冲突；数据库迁移是单一链路，并行开发常撞车。

部署

Docker Compose 一套镜像跑全部，web 与发文 worker 共用同一镜像。发文依赖浏览器自动化（Linux 专属），且必须单实例运行。

2 核心决策：粗粒度，而非「微服务全家桶」

这是本方案与「直接上 12 个 Go 微服务」最大的区别，也是最需要老板拍板的地方。

⚠️ 为什么不建议一步到位 12 微服务

十几个微服务会同时引入：消息总线、分布式锁、跨服务事务、十几条流水线、十几个镜像、还要用 Go 重写包括复杂富文本转换在内的所有逻辑。这套「分布式税」对 2~3 人团队是净负债，且每个跨服务调用都新增一种出错可能。投入大、风险高、见效慢。

✅ 推荐：粗粒度 + 按需抽离

沿代码真实的「断裂面」切成 4~5 个单元，而不是沿目录硬拆。把强耦合的模块留在同一单元里同库同事务（最省心），把天然独立的（发文、热榜、身份）先拆出去。每一步都能独立上线、交付价值、可回滚。需要更细时再继续抽。

3 目标架构：4~5 个可独立部署单元

按真实耦合划分；每个单元有清晰的负责人与边界。

内容核心 保留 Python
文章 / AI 生文 / 工作流编排 / 提示词 / 图片库 / 审核。这些共享「文章状态机+富文本」，强行拆开就是制造分布式事务，故同库同事务。
发文域 保留 Python
发布任务 / 账号 / 浏览器驱动 / 发布 worker。绑定 Linux 浏览器自动化、加密凭据、单实例约束——天然独立，第一阶段即可独立部署。
基础身份 可 Go
用户 / 平台 / 心跳。全员依赖，先稳定契约。
热榜 已独立
第三方热榜聚合，现状已是独立 Node 子服务，零成本。
网关 / 聚合 可选
统一入口、鉴权、路由，作为渐进迁移的「接缝」。
保留 Python 复杂事务/浏览器绑定，短期不重写 可 Go 低耦合域，适合用 Go 练手并受益 可选 按需引入

Go 语言的定位

不是「全量重写」，而是按性能与团队边界按需抽离。身份、热榜、数据统计这类近乎纯增删查的低耦合域，是用 Go 练手并受益的首选；内容核心与发文域的复杂事务短期保留 Python 最稳。把「Go 微服务」当作可达的终态，而不是起步的硬要求。

4 三个诉求如何满足

逐一回应最初提出的三点目标。

① 业务模块相互独立部署，方便维护

发文域第一阶段就独立部署、独立发版、独立扩缩，发文团队完全自治。生文/编排先在「内容核心」内逻辑独立、留好抽离接口；当生文确实需要单独扩容时，再沿这一个接口抽成独立单元。先模块化、保留接缝、按需再抽

② 前端菜单页独立 URL 路由 + 是否引入 ant-design-pro

现状是「无路由单页」，地址栏永远不变。建议分两步：

步骤做什么工期 / 风险
第一步
推荐先做		引入路由库（react-router），保留现有全部组件。把每个菜单映射成唯一 URL（如 /content/pending），可直达、可分享、可加书签——直接满足诉求②。约 2~3 周
中等
第二步
按需评估		是否上 ant-design-pro。它能省掉大量列表/表单代码、自带后台皮肤与权限模型；但属重型框架，富文本编辑器、节点编辑器等自研组件仍需保留，且全量迁移工期长。约 5~8 周
较高

对 ant-design-pro 的建议：不要为了「换框架」而换。 如果主要诉求是「表单/列表更规整 + 现成后台模板」，可以只按需引入 antd 组件库，路由用轻量方案即可，无需迁到 UmiJS 全家桶。等第一步上线后用真实痛点再决定是否深入。

③ 多人协作如何用 Git 避免冲突

团队已有良好的提交规范与 PR 流程。避免冲突的真正杠杆有四个：

措施作用
保持单一代码仓库
（monorepo）		2~3 人用多仓库是净负担。一个仓库 + 按单元分目录，统一管理最省心。
补「代码归属」配置
（CODEOWNERS，当前缺）		核心手段：人按目录分工 → 改的是不同文件 → 天然不撞车。冲突只剩「公共契约 / 共享代码」，而那本就该多人评审。
主干开发 + 短命分支沿用现有 PR 流程，特性分支控制在 2~3 天内合并，避免长命大分支积累冲突。
拆分数据库迁移链当前单一迁移链是并行开发频繁撞车的隐藏根源。每个单元各管自己的迁移，互不干扰。

5 迁移路线（每步独立上线、可回滚）

不追求「一次切完」，追求「每步交付价值」。

  1. 地基（不动业务）：补代码归属配置 + 拆迁移链；前端引入路由打通独立 URL。✓ 立即交付诉求②，零业务风险
  2. 发文域独立：把发文相关模块整体打包成独立部署单元（保留 Python，不重写），仅改三处对接点。✓ 交付诉求①核心，发文团队自治
  3. 内容核心收口：把文章状态机统一成单一写入口 + 对外原子接口；内容与发文之间通过契约/事件清晰解耦。✓ 两大单元彻底分清边界
  4. 按需 Go 化：身份、热榜、数据统计等低耦合域按团队意愿用 Go 抽离、练手。✓ 渐进引入 Go，风险可控
  5. 按需扩展：哪个单元真遇到性能瓶颈，才为它付「多实例 + 物理拆库」的成本；前端按需评估 ant-design-pro。✓ 只在真有需要时投入

6 主要风险与应对

风险应对
过度拆分吃掉开发产能本方案核心对策：粗粒度起步、模块化单体、按需再抽，不一次上微服务全家桶。
内容与发文双向依赖一个方向用「一次性打包数据」的接口拉取，另一方向用事件 + 本地只读副本，避免互相直连数据库。
跨模块的「同一笔事务」把文章状态的写入收归唯一入口，消灭多方竞争；强耦合的逻辑保持在同一单元内不拆。
前端页面状态在路由切换后丢失把筛选条件放进 URL（反而更好用、可分享），编辑器加未保存拦截，实时连接在切页时正确释放。
迁移周期偏长绞杀者式迁移天然渐进；每个阶段独立交付价值（独立 URL、发文自治先见效），允许新旧长期共存。

7 需要老板拍板的决策

下面每一项给出推荐选项，确认后即可启动。

  1. A. 拆分粒度：粗粒度（4~5 单元）还是细粒度（10+ 微服务）？
    ✅ 推荐：粗粒度起步、按需再细分。
    备选：直接上细粒度微服务——投入与风险显著更高，不建议当前规模采用。
  2. B. Go 的引入节奏：终态可选 vs 起步即全量 Go？
    ✅ 推荐：低耦合域先用 Go 练手，核心域短期保留 Python，按需推进。
    备选：全量 Go 重写——需评估重写成本与人力储备。
  3. C. 前端：先打通独立 URL 路由，ant-design-pro 按需再评估？
    ✅ 推荐：第一步轻量路由化（2~3 周，直接满足诉求②）；是否上 Pro 上线后再定。
    备选：一步到位迁 ant-design-pro 全家桶（5~8 周，较高风险）。
  4. D. 是否立即启动两项「零风险地基」？
    ✅ 推荐：现在就做——① 补代码归属配置（CODEOWNERS）② 拆数据库迁移链。立竿见影减少多人冲突，无业务风险。
