## 1. 背景与问题陈述

### 1.1 业务背景

内容运营团队需要把同一批文章（游戏GEO软文）**分发到多个内容平台**，以获取自然流量与内容曝光。每个平台都有独立的：

- 登录态（账号 + Cookie，且会过期）；
- 编辑器（富文本结构、封面规则、小标题样式各不相同）；
- 发布流程（预览、二次确认、风控验证码）。

### 1.2 核心痛点（按工时与风险排序）

| # | 痛点 | 现状成本 | 影响 |
|---|------|----------|------|
| P1 | **重复手工搬运**：一篇文章发到 N 个平台 = N 次"复制→粘贴→排版→传封面→发布" | 单篇单平台约 5–10 分钟，纯体力 | 运营时间被低价值劳动占满 |
| P2 | **多账号登录态管理混乱**：账号多、Cookie 易过期 | 失败后才补登录，事后救火 | 发布成功率不可控 |
| P3 | **平台风控/验证码**：自动化中途弹验证码、登录失效 | 全自动方案脆弱 | 难以稳定无人值守 |
| P4 | **排版与配图耗时**：平台要求小标题、配图、封面 | 每篇额外数分钟 | 产能瓶颈 |
| P5 | **内容生产本身慢**：选题→成稿链路长 | 人工写作 | 上游供给不足 |
| P6 | **缺乏过程留痕**：谁在何时发了什么、失败原因 | 无统一记录 | 协作与排障困难 |

### 1.3 调研与差异化定位

| 方案类型 | 代表 | 优点 | 与本项目的差异 |
|----------|------|------|----------------|
| 平台官方「同步助手」 | 各平台一键多发 | 官方稳定 | 跨厂商不互通、不可编排账号矩阵、无自有内容库 |
| 第三方"矩阵发布"SaaS | 蚁小二 / 简媒等 | 多平台 | 数据出域、定制受限、无法接入自有 AI 生文与图片库 |
| RPA / 纯脚本 | Selenium 脚本 | 灵活 | 验证码即崩、无人工接管、无任务编排与留痕 |

**差异化定位**：本项目走"**自动化为主 + 人工接管兜底（noVNC）**"路线 —— 浏览器自动化覆盖大部分流程，遇风控/失效时把同一个浏览器会话交给人远程处理，而非整单失败，直接回应 P3。

## 2. 产品目标

1. 把"单篇单平台发布"从分钟级手工降为一次配置、批量执行。
2. 统一管理多平台账号登录态，支持续登与状态检查。
3. 提供发布失败的人工接管通道，遇验证码/失效时由人处理后继续。
4. 内置 AI 生文，打通"选题库 → 成稿 → 入库 → 发布"链路，解决上游供给（P5）。
5. 全流程留痕：任务/记录状态、日志、审计、失败截图。

## 3. 用户画像（Persona）

### P-A 内容运营 / 小编（主用户，role=operator）

- **目标**：把 N 篇稿子发到若干平台账号，越省事越好。
- **能力**：会用富文本编辑器，不懂技术。
- **关键诉求**：批量、可预览分配、失败有提醒、只处理验证码这类必须人工的环节。

### P-B 运营主管 / 系统管理员（role=admin）

- **目标**：管理账号矩阵与团队成员，掌握整体发布健康度，能审计追责。
- **关键诉求**：账号授权导入导出、用户管理、操作日志、系统状态总览。

### P-C 远程接管操作者（可能与 P-A 同人）

- **目标**：收到"需要人工"信号后，通过 noVNC 打开正卡住的浏览器，扫码/过验证码，让流程继续。
- **关键诉求**：链接直达、看得到实时页面、操作完一键确认。

## 4. 核心用户场景与用户故事

> 格式：作为 _角色_，我想要 _目标_，以便 _价值_。每条附**验收要点**与**对应实现**。

### S1 批量多平台发布（核心主线）

> 作为运营，我想把"一组文章"轮流分发到"多个账号"，一次提交后台自动执行。

- 验收：可创建 `group_round_robin` 任务，选文章分组 + 多账号；提交前可**预览分配**；执行后每个(文章,账号)落一条可追踪记录。
- 实现：`POST /api/tasks`、`POST /api/tasks/preview`、`POST /api/tasks/{id}/execute`；轮询分配 `account_index = article_index % N`（`tasks/service.py`）。

### S2 单篇定向发布

> 作为运营，我想把"某一篇"发到"某一个号"，用于补发或测试。

- 验收：`single` 任务类型，指定 `article_id` + 账号。
- 实现：`task_type="single"`，默认平台 `toutiao`。

### S3 发布中人工接管（验证码 / 登录失效）

> 作为运营，当自动化遇到验证码或登录态失效时，我想用浏览器远程接管同一会话处理，而不是整单失败。

- 验收：记录进入 `waiting_user_input`，返回 noVNC URL；人工处理后调用 resolve 接口，记录重置 `pending` 并继续。
- 实现：驱动抛 `UserInputRequired` → `runner.py` 附 `novnc_url` → `POST /api/publish-records/{id}/resolve-user-input`。

### S4 发布前停顿人工确认

> 作为运营，对重要稿件我想"自动填好但不点发布"，由我预览后再确认。

- 验收：`stop_before_publish=true` 时记录停在 `waiting_manual_publish`，人工 `manual-confirm` 决定成功/失败。
- 实现：`POST /api/publish-records/{id}/manual-confirm`。

### S5 多账号授权与续期

> 作为运营，我想新增平台账号并在登录态过期时续登。

- 验收：发起登录会话→noVNC 扫码登录→系统抓取 `storage_state.json`→账号状态 `valid`；过期可 relogin / check。
- 实现：账号登录会话状态机（`accounts/auth.py`），`POST /api/accounts/{id}/login-session`、`/check`、`/relogin`。

### S6 AI 生文入库

> 作为运营，我想从选题库批量生成文章并直接入库，缩短成稿时间。

- 验收：选 Skill + Prompt 模板 + 选题（手动选 or 自动按分类轮取），后台并发生成（max_workers=4），完成后文章落 `articles` 表。
- 实现：`POST /api/generation/sessions`，LangGraph 管线（`ai_generation/pipeline.py`），飞书多维表选题同步。

### S7 异常告警

> 作为运营/主管，任务跑完（尤其有失败）我想第一时间在飞书收到通知。

- 验收：任务进入终态（succeeded/partial_failed/failed/cancelled）即推送飞书卡片，含成功/失败/总数。
- 实现：`shared/feishu.py:notify_task_finished`，由 `aggregate_task_status` 触发（任务完成即秒级触达）。

### S8 操作留痕与审计

> 作为主管，我想审计谁在何时做了什么（登录、建账号、改密码等），用于追责。

- 验收：`/api/audit-logs`（admin only）按 user/action/target/时间过滤，游标分页，密码等敏感字段自动脱敏。

### S9 系统状态总览

> 作为主管，我想看到平台整体健康度。

- 验收：`/api/system/status` 返回文章数、账号数、任务数、待执行任务数、活跃浏览器会话数、worker 在线、浏览器/noVNC 运行时就绪。
- 实现：`system/system_router.py:read_system_status`（admin only）。

## 5. 价值假设（可证伪）

> 用"假设 — 度量 — 证伪条件"表达，便于上线后用数据检验。

| # | 价值假设 | 度量指标 | 证伪条件 |
|---|----------|----------|----------|
| H1 | 自动化能把单篇多平台发布的人工耗时从分钟级降到秒级配置 | 单篇单平台人工操作时长 | 自动化后人工仍 > 2 分钟/篇 |
| H2 | "自动化 + 人工接管"能把发布成功率稳定在高位 | 发布成功率（succeeded / 总记录） | 成功率长期偏低且接管也救不回 |
| H3 | 异常及时触达能缩短故障发现时间 | 失败到运营知晓的时延 | 失败后长时间才被发现 |
| H4 | AI 生文能显著缩短成稿时间、提升上游供给 | 篇均成稿时长、日产能 | AI 稿返工率过高，净效率未提升 |
| H5 | 统一账号矩阵管理能降低登录态失效导致的失败 | 因登录态失效的失败占比 | 失效仍是失败主因 |

## 6. 关键指标（North Star + 指标体系）

### 6.1 北极星指标

> **每周成功发布记录数（succeeded PublishRecord / 周）** —— 直接代表"平台为运营创造的分发产能"。

### 6.2 一级指标（健康度）

| 指标 | 数据来源 |
|------|----------|
| 发布成功率 | `publish_records.status` 聚合（succeeded / 总记录） |
| 异常触达时延 | 任务终态 → 飞书推送时间 |
| 人工接管率 | `waiting_user_input` 记录占比 |
| worker 在线率 | `WorkerHeartbeat`（30s 心跳） |

### 6.3 二级指标（效率 / 体验 / 质量）

- **效率**：篇均自动化耗时、批量任务平均时长、AI 生文篇均时长。
- **质量**：发布失败 TOP 原因分布（登录失效 / 验证码 / 平台报错 / 网络）、AI 稿返工率。
- **体验**：从"发起接管"到"处理完成"的平均时长。

## 7. 功能范围

本期交付的功能范围：

- 文章 CRUD 与状态管理、文章分组、资源上传（小文件 + 分块）。
- 批量发布（分组轮询）与单篇发布，提交前分配预览。
- 人工接管（验证码/失效）与发布前确认闭环。
- 多账号授权、续登、状态检查、授权导入导出。
- 任务终态飞书告警，操作审计。
- AI 生文批量入库、AI 智能排版、图片库与配图。
- 系统状态总览。

---

## 8. 约束、假设与依赖

- **平台约束**：发布依赖 `Xvfb / x11vnc / websockify / noVNC`，在 Docker 镜像内运行；发布 worker 单实例。
- **并发约束**：全局并发发布上限 5（`GEO_PUBLISH_MAX_CONCURRENT_RECORDS`），每账号串行，降低平台风控触发。
- **外部依赖**：MySQL 8（ngram 全文检索）、MinIO（图片库）、LiteLLM 可达的 AI 服务、飞书（webhook 告警 + 多维表选题，两套不同凭据）。
- **合规假设**：账号授权由运营本人提供；noVNC 默认只绑 `127.0.0.1`，远程访问走 VPN/SSH 隧道。
