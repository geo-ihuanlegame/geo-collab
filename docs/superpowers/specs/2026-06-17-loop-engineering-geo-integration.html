一页摘要

这是什么
Loop Engineering（循环工程）是 2026 年 6 月在 AI 工程圈被 Addy Osmani / Boris Cherny 集中提出的新范式：不再坐在 Agent 旁边喂 prompt，而是给它一个外循环——目标 / 上下文 / 工具 / 评估 / 停止条件五件套，让它自己跑到达标为止。Anthropic 自家 Claude Code 已内置 /loop /goal /batch 命令。
和 GEO 关系
GEO 的 pipeline 节点注册表 / 驱动注册表 / 问题池 / 方案池 / worker 状态机 / 飞书 + 热榜集成已经覆盖了 70% 的 Loop 基础设施，缺的是评估器 + 反馈回流 + 回路 DAG 三块拼图。
最痛的痛点
当前 GEO 上游选题/选词是人工录飞书文档，实时性差、准确度无客观评价——这恰好是 Loop Engineering 教科书级的落点。
双轨方案
方案 A（1-2 个月跑通）：三段挂载 Loop——选题 / 评审 / 回流，立刻能讲数字；方案 C（3-6 个月愿景）：升级为 Agent Town，运营只设目标和预算，行业最前沿叙事。
要拍板的事
这周拍板：是否同意花 2 周 × 1 个工程师 做一个"选题 Loop POC"，把热门词从手工录飞书 → Agent 自动抓 + 自评 + 写回。这是把 Loop 引入 GEO 的最小可信切入点。

Section 1 / 9 · 什么是 Loop Engineering：五件套

从「会写 prompt」升级为「会设计 Loop」——这是 2026 年 AI 工程圈的新分水岭

一、Prompt Engineering 时代留下来的三个痛

prompt 全靠人
同一个需求一个人改五版、一个不改也能用，质量没标准、不可复制。
Agent 干一次就停
错了要人发现、人纠正、再让它重来——人始终在循环里。
上下文越聊越脏
同一个对话窗越走越偏，最后只能开新窗——浪费 token、丢失进度。

二、Loop Engineering 的五件套（缺一不可）

"找出 20 个高潜选题"

04

自迭代

4产出评估
5停止条件

五件套缺一不可 · 顺时针环回 · 达标则出环，否则回到 ① 重新跑下一轮

三、Prompt Engineering vs Loop Engineering

Prompt Engineering 时代

  • 人是 Agent 的司机——每步都打方向盘
  • 一次调用产一个结果，错了人来纠
  • prompt 是核心资产，分散在人手里
  • Agent 越长跑越偏，最后只能重启
→

Loop Engineering 时代

  • 人是系统设计者——设目标、定刹车、看结果
  • Loop 跑到达标为止，Agent 自评自纠
  • Loop 配方是核心资产，沉淀在系统里
  • 每轮重置上下文，状态留在 DB / 磁盘

关键澄清：Loop 不是替代 Prompt，而是上层抽象——一个 Loop 里有多个 Prompt，prompt 烂照样产烂活、只是产得更快。这就是"从 Prompt Engineer 升级为 Meta-Prompt Engineer"的含义。

三、缺的三块拼图（红区，本次方案的核心增量）

🚦
评估器
目前全链路只有"人工审核"一个闸，AI 自己说不出"这篇不好、要重写"。Loop 的"评估"靠它。

🔄

↩️
回路 DAG
Pipeline 当前严格线性（node_index 递增），没有"评估不达标 → 回上节点重试"的环。

核心判断：补齐这三块拼图，GEO 就从「自动化平台」升级成「自迭代平台」。不需要推倒重来，每块拼图都有现成抓手。

Section 5 / 9 · 方案 C · Agent Town 愿景

把 pipeline 从「执行步骤」升级成「任务规格」——运营只声明目标，一组常驻 Agent 自己分工跑

"运营成本从 每人盯 3 个账号，降到 每人定 30 个账号的策略"——这是 Agent Town 给老板的内部叙事，也是对外（融资 / 客户）的故事主轴。
— 本方案的业务承诺；行业对标：明略 Octo / Steve Yegge Gas Town

一、运营的工作变成什么样

每周一在 GEO 后台设目标：「这周产 300 篇过审 + 分发到这 12 个账号，预算 500 元 API 费用」；设几个人工 checkpoint（"发布前必须我点确认" 或者全自动）；周五看一份系统自动产出的「本周复盘 + 下周建议」。

必须配的「刹车」机制（缺一不可）

  • 预算闸：每个 Agent 每天 API 花费有上限，超了自动停。
  • 强制人工 checkpoint：发布 / 删账号 / 调主流量分发策略 = 必须人工二次确认。
  • 行为黑匣子：所有 Agent 决策走 AuditLog，随时回溯。
  • 沙盒账号优先：探索新平台先用沙盒账号小流量跑，达标才动主账号。

二、要让 A 不挡 B 的"扩展空间"清单

实现方案 A 时，必须留好下面这些口子（成本极低、不留就要回头改）：

  • 评审 Loop 的 evaluator / refiner 用 pipeline 节点形态而不是硬编码，注册到 nodes/__init__.py
  • flow_meta 字段预留 loop_back_to: node_id 配置（用不上时空着），不要写死成线性
  • 评估器的评分维度配置化（事实 / 可读 / 风格 / 合规 + 用户可加），不要硬编码 4 维
  • 所有 Loop 节点都带 max_iterations 和 timeout——为了 A 不发散，也为了 B/C 时不需要重写
  • metrics_collector 写入的 Article.metrics 用 JSON 字段，未来加新指标不写迁移
  • 评审 / 回流 Loop 的状态机 schema 跟 PublishRecord 对齐（success / failed / cancelled / waiting），方便 C 阶段统一抽象

方案 B 不需要单独立项——只要 A 实现时遵守上面 6 条，B 就是 A 上线后的自然形态。讨论稿对老板讲 A + C 即可。

Section 9 / 9 · 决策点与开放问题

老板要拍板的事 · 内部要对齐的事 · 留给后续讨论的事

一、立刻能做的（这周或下周）

需要拍板的事

  • 是否同意 方案 A 起步 + 方案 C 作为愿景 的双轨路线
  • 是否同意花 2 周 × 1 个工程师 做选题 Loop POC
  • Agent 的 API 月度预算 上限是多少（直接决定 Loop 并发 + 最大轮次）
  • 方案A 到 C 的规划方向是否ok？

需要内部对齐的事

  • 已有同事在做选题这块——并入方案 A 还是双轨独立？需要先同步
  • 评估器的"打分维度 + 评分模型 + 提示词初稿"——需要 专家参与定义，不是纯技术活
  • 方案 A 全套 2-3 人月，现有团队能挤出来吗？是否需要 新招 1 人专门做 Agent / Loop 方向
  • Loop 评分模型用 GEO_AI_FORMAT_MODEL (deepseek 系列) 还是单独配？影响成本 5-10x

二、POC 验收标准（2 周后这样验收）

选题 Loop POC · Definition of Done

  • 选题 Loop 每小时自动跑一次，不需要人工触发
  • 飞书问题池里 至少 80% 新增条目来自 Agent 自动抓取 + 评分
  • 每条新增条目带 0-100 评分 + 评分理由文本
  • 运营可以在飞书表里"否决 / 通过"，反馈写回 Loop 让评估器调整权重
  • 上线 1 周后产出第一份"自动选题命中率"报表（被采用的词 / 总抓取数）
  • 所有 Agent 决策走 audit_logs，运营随时回溯

三、留给后续讨论的开放问题

这些问题不影响 POC 启动，但 W6-W9 期间需要回答

  • 评审 Loop 的 evaluator 是否要支持多模型投票（同一篇文章用 3 个模型评分取均值）——质量更稳但成本 ×3
  • 回流 Loop 的平台数据源哪些是 API 直接拉、哪些是半自动飞书表录入？头条统计 / 公众号 stat / 抖音开放平台分别是什么权限要求
  • Agent Town 阶段的消息总线 / 任务池用什么实现——直接复用现在的 worker + DB 乐观锁？还是引入 Redis Stream / RabbitMQ？
  • Skill 层激活时，是否同时引入 MCP 作为内外统一接口？还是 Skill 走内部、MCP 仅对外
  • Agent Town 的 UI 是改造现有"智能体管理" tab 还是新建 tab？这影响前端工作量估算
⟳ 起步建议：2 周 + 1 个工程师 + 选题 Loop POC
所有论证都不如一个能跑起来的 demo——这是把 Loop Engineering 引入 GEO 的最小可信切入点。POC 验收通过则继续 A.2 / A.3；POC 失败也已经验证了"AI 评分 + 飞书写回"链路，本身是 Agent Town 的预热。低成本、高信息密度、不可逆性低。
